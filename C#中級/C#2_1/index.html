    カプセル化の基本原則

    カプセル化の本質

    カプセル化とは「関連するデータと処理をまとめて、外部からの不適切なアクセスを制御する」仕組みです。

    身近な例で理解する

    • ATM機：内部の現金や処理回路は隠されており、決められた操作（入金・出金ボタン）でのみアクセス可能
    • スマートフォン：内部回路は隠されており、画面とボタンという決められたインターフェースでのみ操作可能
    • 自動車：エンジンの詳細は隠されており、ハンドル・アクセル・ブレーキという決められた方法でのみ制御可能
    // 悪い例：カプセル化されていない
class BadBankAccount {
    public double balance;  // 外部から直接アクセス可能
}

BadBankAccount account = new BadBankAccount();
account.balance = -1000;  // 不正な値でも設定できてしまう

// 良い例：カプセル化されている
class GoodBankAccount {
    private double balance;  // 外部からアクセス不可

    public double getBalance() {  // 読み取りのみ許可
        return balance;
    }

    public boolean deposit(double amount) {  // 制御された方法でのみ変更可能
        if (amount &gt; 0) {
            balance += amount;
            return true;
        }
        return false;
    }
}

    カプセル化の3つの利点

    • データ保護：不正な値の設定を防ぐ
    • 内部実装の隠蔽：クラスの使用者は内部の詳細を知る必要がない
    • 変更の局所化：内部実装を変更しても外部に影響しない

    getter/setterの高度な活用パターン

    1. 計算プロパティ（Computed Property）

    実際にはフィールドを持たず、毎回計算して値を返すgetter

    class Circle {
    private double radius;

    public double getRadius() {
        return radius;
    }

    public void setRadius(double radius) {
        if (radius &lt; 0) {
            throw new IllegalArgumentException(&quot;半径は負の値にできません&quot;);
        }
        this.radius = radius;
    }

    // 計算プロパティ：毎回計算する
    public double getArea() {
        return Math.PI * radius * radius;
    }

    public double getCircumference() {
        return 2 * Math.PI * radius;
    }
}

    2. 遅延評価プロパティ（Lazy Property）

    初回アクセス時にのみ計算し、その後はキャッシュされた値を返す

    class ExpensiveCalculation {
    private Double cachedResult = null;

    public double getResult() {
        if (cachedResult == null) {
            System.out.println(&quot;重い計算を実行中...&quot;);
            try {
                Thread.sleep(1000); // 重い処理をシミュレート
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
            cachedResult = complexCalculation();
        }
        return cachedResult;
    }

    private double complexCalculation() {
        // 複雑な計算処理
        return Math.random() * 1000;
    }
}

    3. 依存プロパティ（Dependent Property）

    他のプロパティが変更されると自動的に再計算される

    class Rectangle {
    private double width;
    private double height;
    private Double cachedArea = null;

    public double getWidth() {
        return width;
    }

    public void setWidth(double width) {
        if (width &lt; 0) {
            throw new IllegalArgumentException(&quot;幅は負の値にできません&quot;);
        }
        this.width = width;
        this.cachedArea = null; // 面積のキャッシュをクリア
    }

    public double getHeight() {
        return height;
    }

    public void setHeight(double height) {
        if (height &lt; 0) {
            throw new IllegalArgumentException(&quot;高さは負の値にできません&quot;);
        }
        this.height = height;
        this.cachedArea = null; // 面積のキャッシュをクリア
    }

    public double getArea() {
        if (cachedArea == null) {
            cachedArea = width * height;
        }
        return cachedArea;
    }
}

    コンストラクタチェーンとは

    基本概念

    コンストラクタチェーンとは、一つのコンストラクタから別のコンストラクタを呼び出す仕組みです。 コードの重複を避け、初期化ロジックを一箇所にまとめることができます。

    構文の説明

    class Product {
    private String name;
    private double price;
    private String category;

    // ① デフォルトコンストラクタ
    public Product() {
        System.out.println(&quot;デフォルトコンストラクタ実行&quot;);
        this.name = &quot;未設定&quot;;
        this.price = 0.0;
        this.category = &quot;その他&quot;;
    }

    // ② this() でコンストラクタ①を先に呼び出す
    public Product(String name, double price) {
        this(); // デフォルトコンストラクタを呼び出し
        System.out.println(&quot;2引数コンストラクタ実行&quot;);
        this.name = name;
        this.price = price;
    }

    // ③ this(name, price) でコンストラクタ②を先に呼び出す
    public Product(String name, double price, String category) {
        this(name, price); // 2引数コンストラクタを呼び出し
        System.out.println(&quot;3引数コンストラクタ実行&quot;);
        this.category = category;
    }

    // getter/setterメソッド
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public double getPrice() { return price; }
    public void setPrice(double price) {
        if (price &lt; 0) {
            throw new IllegalArgumentException(&quot;価格は0以上である必要があります&quot;);
        }
        this.price = price;
    }

    public String getCategory() { return category; }
    public void setCategory(String category) { this.category = category; }
}

    実行順序

    Product product = new Product(&quot;ノートPC&quot;, 80000, &quot;電子機器&quot;);

// 実行順序：
// 1. デフォルトコンストラクタ実行
// 2. 2引数コンストラクタ実行
// 3. 3引数コンストラクタ実行

    メリット

    • 重複コードの削減：共通の初期化処理を一箇所にまとめる
    • 保守性の向上：初期化ロジックの変更が一箇所で済む
    • バグの軽減：初期化忘れを防ぐ

    staticメンバーとインスタンスメンバー

    staticの理解

    staticメンバーは、クラスに属し、すべてのインスタンスで共有されます。

    class Counter {
    private static int totalCount = 0;  // すべてのインスタンスで共有
    private int instanceCount = 0;      // インスタンス固有

    public Counter() {
        totalCount++;      // クラス全体のカウンタ
        instanceCount++;   // このインスタンスのカウンタ
    }

    public static int getTotalCount() {
        return totalCount;
    }

    public int getInstanceCount() {
        return instanceCount;
    }

    public void increment() {
        instanceCount++;
        totalCount++;
    }

    // static初期化ブロック
    static {
        System.out.println(&quot;Counterクラスが初回ロードされました&quot;);
        totalCount = 0;
    }
}

    staticメソッドのユーティリティクラス例

    class MathUtils {
    // インスタンス化を防ぐためのprivateコンストラクタ
    private MathUtils() {
        throw new UnsupportedOperationException(&quot;ユーティリティクラスはインスタンス化できません&quot;);
    }

    public static double calculateDistance(double x1, double y1, double x2, double y2) {
        return Math.sqrt(Math.pow(x2 - x1, 2) + Math.pow(y2 - y1, 2));
    }

    public static boolean isPrime(int number) {
        if (number &lt; 2) return false;
        for (int i = 2; i &lt;= Math.sqrt(number); i++) {
            if (number % i == 0) return false;
        }
        return true;
    }

    public static int factorial(int n) {
        if (n &lt; 0) throw new IllegalArgumentException(&quot;負の数の階乗は計算できません&quot;);
        if (n &lt;= 1) return 1;
        return n * factorial(n - 1);
    }
}

    }

        return accountHolder;
    }

    public void setAccountHolder(String accountHolder) {
        if (accountHolder == null || accountHolder.trim().isEmpty()) {
            throw new IllegalArgumentException(&quot;口座名義は空にできません&quot;);
        }
        this.accountHolder = accountHolder;
    }

    public double getBalance() {
        return balance;
    }

    private void setBalance(double balance) {
        if (balance &lt; 0) {
            throw new IllegalArgumentException(&quot;残高は負の値にできません&quot;);
        }
        this.balance = balance;
    }

    public boolean deposit(double amount) {
        if (amount &lt;= 0) {
            System.out.println(&quot;入金額は正の値である必要があります&quot;);
            return false;
        }

        this.balance += amount;
        System.out.printf(&quot;%.0f円を入金しました。現在の残高: %.0f円%n&quot;, amount, balance);
        return true;
    }

    public boolean withdraw(double amount) {
        if (amount &lt;= 0) {
            System.out.println(&quot;出金額は正の値である必要があります&quot;);
            return false;
        }

        if (amount &gt; balance) {
            System.out.printf(&quot;残高が不足しています。現在の残高: %.0f円%n&quot;, balance);
            return false;
        }

        this.balance -= amount;
        System.out.printf(&quot;%.0f円を出金しました。現在の残高: %.0f円%n&quot;, amount, balance);
        return true;
    }

    public String getAccountInfo() {
        return String.format(&quot;口座番号: %s%n名義人: %s%n残高: %.0f円&quot;,
                           accountNumber, accountHolder, balance);
    }

    実践例題

    例題1: 図書管理システムのBookクラス設計

    要求仕様： - 書籍のタイトル、著者、ISBN、価格を管理 - ISBNは一度設定したら変更不可 - 価格は0以上でなければならない - 書籍の情報を文字列で出力するメソッドを提供

    解答例： 

    class Book {
    private String title;
    private String author;
    private final String isbn;  // finalで変更不可
    private double price;

    public Book(String title, String author, String isbn, double price) {
        this.title = title;
        this.author = author;
        this.isbn = isbn;
        setPrice(price); // バリデーション付きsetterを使用
    }

    // getter/setterメソッド
    public String getTitle() {
        return title;
    }

    public void setTitle(String title) {
        this.title = title;
    }

    public String getAuthor() {
        return author;
    }

    public void setAuthor(String author) {
        this.author = author;
    }

    public String getIsbn() {
        return isbn; // setterは提供しない（final）
    }

    public double getPrice() {
        return price;
    }

    public void setPrice(double price) {
        if (price &lt; 0) {
            throw new IllegalArgumentException(&quot;価格は0以上である必要があります&quot;);
        }
        this.price = price;
    }

    // 書籍情報を文字列で返すメソッド
    public String getBookInfo() {
        return String.format(&quot;タイトル: %s%n著者: %s%nISBN: %s%n価格: %.0f円&quot;,
                           title, author, isbn, price);
    }

    // 割引価格を計算するメソッド
    public double getDiscountedPrice(double discountPercentage) {
        if (discountPercentage &lt; 0 || discountPercentage &gt; 100) {
            throw new IllegalArgumentException(&quot;割引率は0-100%の間で指定してください&quot;);
        }

        return price * (1 - discountPercentage / 100);
    }

    @Override
    public String toString() {
        return getBookInfo();
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;

        Book book = (Book) obj;
        return isbn.equals(book.isbn);  // ISBNで書籍を識別
    }

    @Override
    public int hashCode() {
        return isbn.hashCode();
    }
}

    重要なポイント

    カプセル化の実践的理解

    • データ保護：privateフィールド + getter/setterでデータへの安全なアクセスを提供
    • 内部実装の隠蔽：クラスの使用者は内部の詳細を知る必要がない
    • 変更の局所化：内部実装を変更しても外部インターフェースに影響しない
    • 責任の明確化：データとそれを操作するメソッドを一箇所にまとめる

    getter/setterの高度な活用パターン

    • 計算プロパティ：実際にはフィールドを持たず、毎回計算して値を返す
    • 遅延評価プロパティ：初回アクセス時にのみ計算し、その後はキャッシュを利用
    • 依存プロパティ：他のプロパティが変更されると自動的に関連データを更新
    • 読み取り専用プロパティ：finalフィールドまたはsetterなしで外部からの変更を防ぐ

    コンストラクタの設計パターン

    • コンストラクタチェーン：this()で共通初期化処理の重複を避ける
    • 実行順序の理解：チェーンされたコンストラクタは呼び出し順序が重要
    • デフォルト値の設定：引数なしコンストラクタで安全な初期状態を保証
    • 必須パラメータの強制：重要なデータはコンストラクタで必須入力にする

    staticの理解と活用

    • クラスレベルのデータ：すべてのインスタンスで共有される
    • ユーティリティメソッド：インスタンス化不要の便利メソッド
    • 初期化ブロック：クラスロード時の一度だけの初期化
    • メモリ効率：インスタンスごとに重複しないデータの管理

    クラス設計の品質指標

    • 単一責任の原則：1つのクラスは1つの明確な責任を持つ
    • データの整合性：不正な状態にならないよう制約を設ける
    • 使いやすさ：直感的で分かりやすいメソッド名とプロパティ名
    • 拡張性：将来の機能追加に対応できる柔軟な設計

    まとめ

    中級クラス設計の本質

    • 安全性の向上：データの整合性を保ち、不正な操作を防ぐ仕組みの構築
    • 保守性の確保：変更に強く、理解しやすいコード構造の実現
    • 再利用性の追求：汎用的で柔軟なクラス設計による開発効率の向上

    初級からの発展ポイント

    • 基本的なフィールド → getter/setterによる制御されたアクセス
    • 単純なコンストラクタ → チェーンとオーバーロードによる柔軟な初期化
    • 基本的なカプセル化 → データ保護と内部実装隠蔽の実践

    実践的なクラス設計の流れ

    1. 責任の明確化：クラスが担う役割を明確に定義
    2. データの特定：管理すべき情報とその制約を洗い出し
    3. 操作の設計：データに対する必要な操作をメソッドとして定義
    4. アクセス制御：適切なpublic/privateの設定でカプセル化を実現
    5. 初期化戦略：コンストラクタで安全で使いやすい初期化を提供

    品質の高いクラスの特徴

    • 明確な責任境界：何をするクラスなのかが一目で分かる
    • 堅牢なデータ保護：不正な状態になることがない
    • 直感的なインターフェース：使う側が迷わない分かりやすさ
    • 将来への対応力：機能追加や変更に柔軟に対応できる構造
