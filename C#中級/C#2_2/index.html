    継承の基本概念

    継承とは何か

    既存のクラス（スーパークラス・親クラス）の特性を受け継いで、新しいクラス（サブクラス・子クラス）を作成する仕組みです。

    現実世界での継承

    継承とは「親から子へ特徴を受け継ぐ」仕組みです。

    • 生物の例：哺乳類 → 犬、猫（哺乳類の特徴を受け継ぎつつ、独自の特徴も持つ）
    • 職業の例：従業員 → 正社員、契約社員（従業員の基本特徴 + 各々の特殊な特徴）
    • 乗り物の例：Vehicle → Car, Airplane（移動するという共通機能 + 各々の移動方法）

    プログラムでの継承の利点

    1. コードの重複排除

    // 継承なしの場合：重複だらけ
class Dog {
    private String name;     // 重複
    private int age;         // 重複
    public void eat() { ... }    // 重複
    public void sleep() { ... }  // 重複
    public void bark() { ... }   // 犬独自
}

class Cat {
    private String name;     // 重複
    private int age;         // 重複
    public void eat() { ... }    // 重複
    public void sleep() { ... }  // 重複
    public void meow() { ... }   // 猫独自
}

// 継承ありの場合：共通部分をまとめる
class Animal {  // スーパークラス
    protected String name;   // 共通（protectedで子クラスからアクセス可能）
    protected int age;       // 共通

    public void eat() { ... }    // 共通
    public void sleep() { ... }  // 共通
}

class Dog extends Animal {  // サブクラス
    public void bark() { ... }   // 犬独自のみ
}

class Cat extends Animal {  // サブクラス
    public void meow() { ... }   // 猫独自のみ
}

    2. 保守性向上

    共通機能の修正が一箇所で済む

    3. 拡張性の向上

    新しい動物クラスを簡単に追加できる

    継承の「IS-A関係」

    継承は「○○は××である」という関係を表現します。 - Dog IS-A Animal（犬は動物である） - Car IS-A Vehicle（車は乗り物である） - Manager IS-A Employee（管理職は従業員である）

    ポリモーフィズムとは

    ポリモーフィズムの語源

    「poly（多くの）」+「morph（形）」= 「多様性」「多態性」
    プログラミングにおいては、「同じインターフェース（メソッド名）で、オブジェクトの実際の型に応じて異なる動作をする」仕組みです。

    基本概念

    同じインターフェース（メソッド名）で、異なる実装を呼び出せる仕組みです。

    身近な例

    • リモコンの「電源ボタン」：
    • テレビ用リモコン → テレビの電源ON/OFF
    • エアコン用リモコン → エアコンの電源ON/OFF
    • 同じ「電源ボタン」だが、実際の動作は機器によって異なる

    プログラムでの利点

    1. 統一的な処理

    // ポリモーフィズムなしの場合
public void processAnimals(List&lt;Object&gt; animals) {
    for (Object animal : animals) {
        if (animal instanceof Dog) {
            ((Dog) animal).bark();
        } else if (animal instanceof Cat) {
            ((Cat) animal).meow();
        } else if (animal instanceof Bird) {
            ((Bird) animal).chirp();
        }
        // 新しい動物が増える度にif文を追加...
    }
}

// ポリモーフィズムありの場合
public void processAnimals(List&lt;Animal&gt; animals) {
    for (Animal animal : animals) {
        animal.makeSound(); // 各動物に応じた鳴き声が自動的に呼ばれる
    }
    // 新しい動物が増えても、この処理は変更不要！
}

    2. 拡張性

    新しいクラスを追加しても、既存のコードを変更する必要がない

    3. 抽象化

    具体的な型を知らなくても、共通のインターフェースで操作できる

    @Overrideアノテーションとメソッドオーバーライド

    @Overrideアノテーション

    子クラスで親クラスのメソッドを「意図的に上書きしています」と宣言する

    class Animal {
    // 親クラスのメソッド
    public void makeSound() {
        System.out.println(&quot;何らかの音&quot;);
    }

    // オーバーライドできないメソッド（final）
    public final void breathe() {
        System.out.println(&quot;呼吸している&quot;);
    }
}

class Dog extends Animal {
    // @Override：親クラスのmakeSoundを置き換え
    @Override
    public void makeSound() {
        System.out.println(&quot;ワンワン&quot;);
    }

    // エラー：breatheはfinalなのでオーバーライド不可
    // @Override
    // public void breathe() { ... }
}

    なぜ@Overrideが重要なのか？

    1. コンパイル時チェック

    class Cat extends Animal {
    // タイポがあってもコンパイラが検出
    @Override
    public void makSound() {  // ←typo: makeSoundの間違い
        System.out.println(&quot;ニャー&quot;);
    }
    // コンパイルエラー：オーバーライドするメソッドが見つからない
}

    2. 意図の明確化

    • @Override：「意図的に親クラスの動作を変更している」
    • コードの可読性向上：他の開発者が意図を理解しやすい

    実行時の動作

    Animal animal = new Dog();
animal.makeSound(); // 実行時にDogクラスのmakeSoundが呼ばれる

// 内部では以下のような処理が行われる：
// 1. animalの実際の型を確認（Dog）
// 2. DogクラスにmakeSoundのOverrideがあるかチェック

    抽象クラスとインターフェース

    基本的な違い

    抽象クラス（Abstract Class）

    • 「不完全なクラス」：一部のメソッドが未実装
    • IS-A関係：「○○は××である」
    • 単一継承：1つの抽象クラスからのみ継承可能
    • 共通実装を持てる：子クラス間で共有するコードを記述可能

    インターフェース（Interface）

    • 「契約書」：「○○ができる」という機能の約束
    • CAN-DO関係：「○○することができる」
    • 多重実装：複数のインターフェースを同時に実装可能
    • 実装は持てない：メソッドの宣言のみ（Java 8以降はdefaultメソッド例外あり）

    使い分けの指針

    抽象クラスを使う場面

    • 共通の基本機能がある場合
    • 子クラス間で共有したいコードがある場合
    • 強い関連性があるクラス群の場合
    // 例：従業員系（共通の基本情報と処理がある）
abstract class Employee {
    protected String name;        // 共通のフィールド
    protected int employeeId;

    public Employee(String name, int employeeId) {
        this.name = name;
        this.employeeId = employeeId;
    }

    public void clockIn() {       // 共通のメソッド
        System.out.println(name + &quot;が出勤しました&quot;);
    }

    // 抽象メソッド：子クラスで異なる実装
    public abstract double calculatePay();
}

    インターフェースを使う場面

    • 機能の追加的な場合
    • 異なる系統のクラスに同じ機能を付与したい場合
    • 複数の機能を組み合わせたい場合
    // 例：描画機能（様々なオブジェクトに描画機能を追加）
interface Drawable {
    void draw();
}

// 図形にも、UIコントロールにも、ゲームオブジェクトにも適用可能
class Circle extends Shape implements Drawable {
    @Override
    public void draw() {
        System.out.println(&quot;円を描画&quot;);
    }
}

class Button extends Component implements Drawable {
    @Override
    public void draw() {
        System.out.println(&quot;ボタンを描画&quot;);
    }
}

class Player extends GameCharacter implements Drawable {
    @Override
    public void draw() {
        System.out.println(&quot;プレイヤーを描画&quot;);
    }
}

    // 抽象クラス：基本的な「動物」の概念
abstract class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public abstract void makeSound();

    public void sleep() {  // 共通実装
        System.out.println(name + &quot;が眠っています&quot;);
    }
}

// インターフェース：「飼えるかどうか」の機能
interface Petable {
    void feed();
    boolean isTrained();
}

// インターフェース：「飛べるかどうか」の機能
interface Flyable {
    void fly();
    int getMaxAltitude();
}

// 組み合わせ例
class Dog extends Animal implements Petable {
    private boolean trained;

    public Dog(String name) {
        super(name);
        this.trained = false;
    }

    @Override
    public void makeSound() {
        System.out.println(name + &quot;：ワンワン！&quot;);
    }

    @Override
    public void feed() {
        System.out.println(name + &quot;にエサを与えました&quot;);
    }

    @Override
    public boolean isTrained() {
        return trained;
    }

    public void train() {
        trained = true;
        System.out.println(name + &quot;をしつけました&quot;);
    }
}

class Bird extends Animal implements Flyable {
    private int maxAltitude;

    public Bird(String name, int maxAltitude) {
        super(name);
        this.maxAltitude = maxAltitude;
    }

    @Override
    public void makeSound() {
        System.out.println(name + &quot;：チュンチュン！&quot;);
    }

    @Override
    public void fly() {
        System.out.println(name + &quot;が飛んでいます&quot;);
    }

    @Override
    public int getMaxAltitude() {
        return maxAltitude;
    }
}

class Parrot extends Animal implements Petable, Flyable {
    private boolean trained;
    private int maxAltitude;

    public Parrot(String name, int maxAltitude) {
        super(name);
        this.maxAltitude = maxAltitude;
        this.trained = false;
    }

    @Override
    public void makeSound() {
        System.out.println(name + &quot;：ハロー！&quot;);
    }

    @Override

        System.out.println(name + &quot;にエサを与えました&quot;);
    }

    @Override

    }

    @Override
    public void fly() {

    }

    public int getMaxAltitude() {
        return maxAltitude;
    }

    superキーワード

    superキーワードとは

    子クラスから親クラスのメンバー（フィールド、メソッド、コンストラクタ）にアクセスするためのキーワードです。

    使用場面1：コンストラクタチェーン

    class Animal {
    protected String name;
    protected int age;

    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }
}

class Dog extends Animal {
    private String breed;

    // super(name, age)で親クラスのコンストラクタを呼び出し
    public Dog(String name, int age, String breed) {
        super(name, age);  // 最初の行で親クラスのコンストラクタを呼び出し
        this.breed = breed; // 子クラス独自の初期化
    }
}

    使用場面2：オーバーライドしたメソッドから親クラスのメソッドを呼び出し

    class Animal {
    public void move() {
        System.out.println(&quot;動物が移動しています&quot;);
    }
}

class Dog extends Animal {
    @Override
    public void move() {
        super.move(); // 親クラスのmoveを先に実行
        System.out.println(&quot;犬が尻尾を振りながら移動しています&quot;);
    }
}

// 実行結果：
// 動物が移動しています
// 犬が尻尾を振りながら移動しています

    使用場面3：親クラスのフィールドへのアクセス

    class Employee {
    protected String name;

    public String getInfo() {
        return &quot;従業員: &quot; + name;
    }
}

class Manager extends Employee {
    @Override
    public String getInfo() {
        return super.getInfo() + &quot; (管理職)&quot;; // 親クラスの情報に追加
    }
}

    実践例題

    例題1: 従業員管理システム

    要求仕様： - 基底クラスEmployeeを作成 - 正社員（FullTimeEmployee）と契約社員（ContractEmployee）を子クラスとして実装 - 給与計算方法が異なる（正社員：月給、契約社員：時給×労働時間） - 全従業員の給与合計を計算する機能

    解答例： 

    import java.time.LocalDate;
import java.time.Period;
import java.util.ArrayList;
import java.util.List;

// 抽象基底クラス
abstract class Employee {
    protected String name;
    protected int employeeId;
    protected LocalDate hireDate;

    protected Employee(String name, int employeeId, LocalDate hireDate) {
        this.name = name;
        this.employeeId = employeeId;
        this.hireDate = hireDate;
    }

    // 抽象メソッド：給与計算
    public abstract double calculateSalary();

    // 勤続年数計算
    public int getYearsOfService() {
        return Period.between(hireDate, LocalDate.now()).getYears();
    }

    public void displayInfo() {
        System.out.printf(&quot;ID: %d, 名前: %s, 勤続年数: %d年%n&quot;,
                         employeeId, name, getYearsOfService());
    }

    // getter
    public String getName() { return name; }
    public int getEmployeeId() { return employeeId; }
    public LocalDate getHireDate() { return hireDate; }
}

// 正社員クラス
class FullTimeEmployee extends Employee {
    private double monthlySalary;
    private double bonus;

    public FullTimeEmployee(String name, int employeeId, LocalDate hireDate, double monthlySalary) {
        super(name, employeeId, hireDate);
        this.monthlySalary = monthlySalary;
        this.bonus = 0;
    }

    @Override
    public double calculateSalary() {
        // 年俸 = 月給 × 12 + ボーナス
        return monthlySalary * 12 + bonus;
    }

    @Override
    public void displayInfo() {
        super.displayInfo();
        System.out.printf(&quot;雇用形態: 正社員, 年俸: %.0f円%n&quot;, calculateSalary());
    }

    public void setBonus(double bonus) {
        this.bonus = bonus;
    }
}

// 契約社員クラス
class ContractEmployee extends Employee {
    private double hourlyRate;
    private int hoursWorked;

    public ContractEmployee(String name, int employeeId, LocalDate hireDate, double hourlyRate) {
        super(name, employeeId, hireDate);
        this.hourlyRate = hourlyRate;
        this.hoursWorked = 0;
    }

    @Override
    public double calculateSalary() {
        return hourlyRate * hoursWorked;
    }

    public void setHoursWorked(int hours) {
        if (hours &lt; 0) {
            throw new IllegalArgumentException(&quot;労働時間は0以上で入力してください&quot;);
        }
        this.hoursWorked = hours;
    }

    @Override
    public void displayInfo() {
        super.displayInfo();
        System.out.printf(&quot;雇用形態: 契約社員, 年収: %.0f円 (時給: %.0f円, 労働時間: %d時間)%n&quot;,
                         calculateSalary(), hourlyRate, hoursWorked);
    }
}

// 従業員管理クラス
class EmployeeManager {
    private List&lt;Employee&gt; employees;

    public EmployeeManager() {
        employees = new ArrayList&lt;&gt;();
    }

    public void addEmployee(Employee employee) {
        employees.add(employee);
    }

    public double getTotalSalary() {
        return employees.stream()
                        .mapToDouble(Employee::calculateSalary)
                        .sum();
    }

    public void displayAllEmployees() {
        System.out.println(&quot;=== 全従業員情報 ===&quot;);
        for (Employee employee : employees) {
            employee.displayInfo();
            System.out.println();
        }
        System.out.printf(&quot;総給与額: %.0f円%n&quot;, getTotalSalary());
    }
}

    重要なポイント

    継承の基本理解

    • IS-A関係：継承は「○○は××である」という関係を表現（Dog IS-A Animal）
    • コードの再利用：共通の機能を親クラスにまとめ、重複を排除
    • 単一継承：Javaでは1つのクラスからのみ継承可能（多重継承は不可）
    • アクセス修飾子の影響：private は継承されない、protected は子クラスからアクセス可能

    ポリモーフィズムの核心概念

    • 動的バインディング：実行時に実際のオブジェクトの型を見てメソッドを決定
    • 統一的なインターフェース：同じメソッド名で異なる実装を呼び出し可能
    • 拡張性の確保：新しいクラス追加時に既存コードの変更が不要
    • 抽象化の実現：具体的な型を知らずに共通操作が可能

    @Overrideアノテーションの重要性

    • @Override：「このメソッドは親クラスのメソッドを意図的に置き換え」という宣言
    • コンパイル時チェック：メソッド名のタイポなどをコンパイル時に検出
    • コードの可読性：意図が明確で、保守しやすいコード
    • リファクタリング安全性：親クラスのメソッド名変更時に子クラスも自動検出

    superキーワードの活用

    • コンストラクタチェーン：super()で親クラスの初期化を呼び出し
    • 機能の拡張：親クラスの処理に追加処理を組み合わせ
    • 重複回避：親クラスの実装を再利用して無駄なコード重複を防ぐ
    • 初期化の保証：親クラスの適切な初期化を確実に実行

    抽象クラスの設計指針

    • 不完全なクラス：一部のメソッドが未実装で、直接インスタンス化不可
    • 共通実装の提供：子クラス間で共有する具体的なコードを含む
    • 強制的な実装：抽象メソッドは子クラスでの実装が必須
    • IS-A関係の強化：より厳密な継承関係を表現

    インターフェースの設計指針

    • 契約の定義：「○○ができる」という機能の約束を明文化
    • 多重実装：複数のインターフェースを同時に実装可能
    • 実装の強制：インターフェースのメンバーは必ず実装が必要
    • CAN-DO関係：異なる系統のクラスに共通機能を付与

    抽象クラス vs インターフェースの使い分け

    • 抽象クラスを選ぶ場面：共通の基本機能がある、強い関連性がある場合
    • インターフェースを選ぶ場面：機能追加的、異なる系統に同じ機能を付与する場合
    • 組み合わせ使用：抽象クラス継承 + 複数インターフェース実装が可能
    • 将来の拡張性：要件変更に応じて適切な選択ができる柔軟性

    まとめ

    継承とポリモーフィズムの本質

    • コードの構造化：関連するクラス群を階層的に整理し、保守性を向上
    • 柔軟性の実現：同じインターフェースで多様な実装を扱える仕組み
    • 拡張性の確保：新機能追加時に既存コードへの影響を最小化

    オブジェクト指向設計の発展

    • 単純なクラス → 継承による階層化 → ポリモーフィズムによる柔軟性
    • 重複コードの排除 → 共通機能の親クラス化 → 個別機能の子クラス化
    • 静的な構造 → 動的な振る舞い → 実行時の型判定と適切なメソッド呼び出し

    実践的な設計パターン

    • テンプレートメソッドパターン：親クラスで処理の流れを定義、子クラスで具体的な処理を実装
    • 戦略パターン：インターフェースで行動を抽象化、具体的な戦略を交換可能に
    • ファクトリーパターン：オブジェクト作成をポリモーフィズムで抽象化
