    レッスン3

    text HTTPを使った具体的なリクエストとレスポンスの仕組みを理解する。ステータスコードの意味、GET/POSTの違い、エラーハンドリングを身近な例で説明し、開発者ツールでの実際の確認方法も含める。開発者ツールの基本操作から実践的なトラブルシューティングまで体系的に習得する。 3.リクエストとレスポンス - データのやり取り # リクエストとレスポンス - データのやり取り

    学習目標

    • リクエスト（要求）とレスポンス（応答）の具体的な内容を理解する
    • ステータスコード（200、404、500など）の意味を知る
    • GET/POSTの基本的な違いを理解する
    • 開発者ツールの基本的な使い方を覚える
    • 実際のブラウザでリクエスト・レスポンスがどこに表示されるかを確認できるようになる

    開発者ツールって何？まず触ってみよう

    1. 開発者ツールとは

    開発者ツールは、ブラウザに標準で搭載されている「Webページの裏側を見る道具」です。普段は見えないHTTP通信の詳細を確認できます。

    怖がらずに触ってみましょう！ - 壊れることはありません - いつでも閉じることができます - Webサイトに影響を与えません

    2. 開発者ツールを開く方法

    Windows/Linux: - F12キーを押す（最も簡単） - Ctrl + Shift + I - 右クリック → 「検証」または「要素を調査」

    Mac: - Cmd + Option + I - 右クリック → 「要素を調査」

    3. 開発者ツールの画面構成

    開発者ツールを開くと、画面が上下または左右に分割されます：

    📘 開発ツールのレイアウト図

    主要なタブ： - Elements: HTMLコードを見る - Network: 通信内容を見る（今回重要！） - Console: エラーメッセージを見る - Sources: JavaScriptコードを見る

    4. 今回使うNetworkタブ

    NetworkタブがHTTP通信を確認する場所です。ここで以下を確認できます： - どのファイルを取得したか - ステータスコード（成功/失敗） - GET/POSTの種類 - 通信にかかった時間 - 送受信したデータの内容

    リクエスト：「情報をください」の詳細

    1. リクエストって具体的に何を送っているの？

    前回学習した通り、ブラウザがサーバーに「このページをください」と要求することをリクエストと呼びました。でも実際には、もっと詳しい情報をやり取りしています。

    📘 HTTPリクエストの詳細構造

    リクエストに含まれる主な情報：

    ① 何が欲しいか（URL）

    GET /products/smartphones.html HTTP/1.1
    - GET = 「取得したい」という意味 - /products/smartphones.html = 欲しいページの場所 - HTTP/1.1 = 使用するHTTPのバージョン

    ② どこのサーバーか（Host）

    Host: www.example.com

    ③ 自分の情報（User-Agent）

    User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 Chrome/120.0.0.0
    - どんなブラウザを使っているか - どんなOSを使っているか

    ④ 受け入れ可能な形式（Accept）

    Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8

    ⑤ 言語設定（Accept-Language）

    Accept-Language: ja,en-US;q=0.9,en;q=0.8

    2. 身近な例で理解しよう

    レストランでの注文と比較：

    📘 レストランの注文例

    リクエストの要素 レストランでの注文 実際のWebリクエスト
    何が欲しいか 「ハンバーガーセットを」 「smartphones.htmlを」
    どこの店か 「○○レストランで」 「www.example.comで」
    自分の情報 「座席番号5番の客です」 「Chromeブラウザです」
    好み・制限 「辛い物は苦手です」 「日本語対応希望」
    支払い方法 「クレジットカードで」 「Cookieあります」

    レスポンス：「情報をお返しします」の詳細

    1. レスポンスって具体的に何が返ってくるの？

    サーバーからブラウザに返されるレスポンスにも、様々な情報が含まれています。

    レスポンスに含まれる主な情報：

    ① 処理結果（ステータスコード）

    HTTP/1.1 200 OK
    - 200 = 成功しました - 404 = ページが見つかりません - 500 = サーバーエラーです

    ② サーバー情報

    Server: Apache/2.4.41
Date: Wed, 29 Jul 2025 10:30:00 GMT

    ③ 内容の形式（Content-Type）

    Content-Type: text/html; charset=UTF-8

    ④ 内容のサイズ（Content-Length）

    Content-Length: 2048

    ⑤ 実際の内容（Body）

    &lt;!DOCTYPE html&gt;
&lt;html&gt;
&lt;head&gt;&lt;title&gt;スマートフォン一覧&lt;/title&gt;&lt;/head&gt;
&lt;body&gt;...&lt;/body&gt;
&lt;/html&gt;

    2. ステータスコードを理解しよう

    ステータスコードは「処理結果の番号」です。レストランでの応答と比較してみましょう：

    2xx系：成功

    • 200 OK「注文の料理をお持ちしました」
    • 201 Created「新しい料理を作成しました」

    3xx系：リダイレクト

    • 301 Moved Permanently「その料理は別のメニューに移りました」
    • 302 Found「一時的に別の場所で提供中です」

    4xx系：クライアントエラー（お客さんの問題）

    • 400 Bad Request「注文内容が理解できません」
    • 401 Unauthorized「会員登録が必要です」
    • 403 Forbidden「そのメニューは注文できません」
    • 404 Not Found「そのメニューは存在しません」

    5xx系：サーバーエラー（お店の問題）

    • 500 Internal Server Error「キッチンで問題が発生しました」
    • 502 Bad Gateway「配達業者と連絡が取れません」
    • 503 Service Unavailable「一時的に営業停止中です」

    3. よく見るエラーの意味

    404 Not Found - 原因：URLが間違っている、ページが削除された - 対処：URLを確認、正しいリンクを探す

    500 Internal Server Error - 原因：サーバー側のプログラムエラー - 対処：時間を置いて再アクセス、管理者に連絡

    403 Forbidden - 原因：アクセス権限がない - 対処：ログインが必要、または権限のないページ

    開発者ツールでの確認方法（詳細版）

    演習1：まず開発者ツールに慣れよう

    Step 1: 開発者ツールを開く 1. 任意のWebサイト（例：https://www.google.com）にアクセス 2. F12キーを押す 3. 開発者ツールが開いたことを確認

    Step 2: Networkタブを見つける 1. 開発者ツール上部のタブを確認 2. 「Network」タブをクリック 3. 最初は何も表示されていません（これは正常です）

    Step 3: ページを再読み込みして通信を記録 1. F5キーでページを再読み込み 2. Networkタブに複数行の情報が表示される 3. これが HTTP通信の一覧です！

    演習2：Networkタブの見方を覚えよう

    Networkタブで表示される情報の場所：

    📘 ネットワークタブの表示

    各列の意味：

    1. Name（名前）
    2. どこを見る: 左端の列
    3. 意味: 取得したファイルの名前やURL

    10. どこを見る: Status の右隣
    11. 意味: ファイルの種類

    14. どこを見る: Type の右隣
    15. 意味: ファイルのサイズ

    16. 例: 25.6 KB, 12.4 KB

    17. Time（時間）

    18. どこを見る: 右端の列
    19. 意味: 通信にかかった時間
    20. 例: 245ms, 89ms

    演習3：GET/POSTメソッドの確認方法

    Method（メソッド）の確認方法：

    1. Networkタブで任意の通信をクリック
    2. 右側または下側に詳細画面が表示される
    3. 「Headers」タブを選択
    4. 「General」セクションを確認

    📘 ネットワークタブのカラム構成

    General セクションで確認できる情報： 

    Request URL: https://www.example.com/page.html
Request Method: GET                    ← ここがメソッド
Status Code: 200 OK                   ← ここがステータスコード
Remote Address: 192.168.1.1:443

    GETメソッドを確認する例： 1. Google で何かを検索 2. 検索結果ページで開発者ツールを開く 3. Networkタブのnameに search を含むリクエストをクリック 4. Headers → General で Request Method: GET を確認 5. URLにパラメータが含まれていることも確認

    POSTメソッドを確認する例（参考）： - ログインフォームがあるサイトでログイン実行 - Networkタブで login に関連するリクエストをクリック - Request Method: POST と表示される - URLにパスワード等が表示されないことを確認

    GET と POST の違い（実際の表示での説明）

    1. 基本的な使い分け

    📘 [GET VS POST] (https://nkaihatsuItDivision.github.io/CodeLingoPicture/WebBasic_3_4.html)

    項目 GET POST
    目的 データを「取得」 データを「送信」
    データの場所 URLに含まれる（アドレスバーで見える） 本文（Body）に含まれる（URLに表示されない）
    安全性 低い（URL表示される） 高い（URL表示されない）
    データサイズ制限 あり（約2KB） なし（大容量可能）
    ブックマーク 可能 不可能
    ブラウザ履歴 残る 残らない

    2. URLでの違いを実際に確認

    GET の例：検索（URLを見てみよう）

    Google で「プログラミング学習」を検索すると、URLが以下のようになります：

    https://www.google.com/search?q=プログラミング学習&amp;hl=ja
                                ↑
                           ここに検索キーワード

    確認方法： 1. Google で検索実行 2. アドレスバーのURLを確認 3. q= の後に検索キーワードが表示されている 4. 開発者ツールのNetworkタブでも Request Method: GET を確認

    📘 URLパラメータ

    POST の例：ログイン（URLを見てみよう）

    ログインフォームでは、パスワードがURLに表示されません：

    https://www.example.com/login
                            ↑
                    パスワードは表示されない

    確認方法（概念的）： 1. ログインフォームのページでアドレスバーを確認 2. ユーザー名・パスワードがURLに含まれないことを確認 3. （開発者ツールのForm DataまたはRequest Bodyで送信内容確認可能）

    3. 開発者ツールでの詳細確認

    GETパラメータの確認： 1. 検索サイトで検索実行 2. NetworkタブでGETリクエストをクリック 3. 「Headers」タブ → 「Query String Parameters」で確認

    Query String Parameters:
q: プログラミング学習
hl: ja

    POSTデータの確認（参考）： 1. ログインフォーム送信 2. NetworkタブでPOSTリクエストをクリック 3. 「Headers」タブ → 「Form Data」または「Request Payload」で確認

    📘 POSTでのForm Data確認

    トラブルシューティング（実践版）

    よくある問題と開発者ツールでの確認方法

    1. 「ページが表示されない」

    確認手順： 1. 開発者ツールのNetworkタブを開く 2. F5で再読み込み 3. Status列を確認： - 何も表示されない → ネットワーク問題 - 404 → URLが間違っている - 500 → サーバー側の問題 - 403 → アクセス権限がない

    2. 「画像が表示されない」

    確認手順： 1. NetworkタブでType列で「image」をフィルタ 2. 赤色（エラー）の画像ファイルを確認 3. 該当ファイルをクリックして詳細確認

    3. 「ページの読み込みが遅い」

    確認手順： 1. NetworkタブでTime列を確認 2. 時間のかかっているファイルを特定 3. Size列でファイルサイズも確認 4. 大きなファイルや遅いファイルが原因を特定

    よくある質問

    Q: 開発者ツールを開くとページが壊れない？ A: 全く問題ありません。開発者ツールは「見るだけ」のツールで、Webサイトに影響を与えません。

    Q: Networkタブに何も表示されない A: ページを再読み込み（F5）してください。開発者ツールを開く前の通信は記録されていません。

    Q: たくさんの通信が表示されて分からない A: 上部のフィルタ（All, Doc, JS, CSS等）を使って絞り込みましょう。最初は「Doc」（ドキュメント）だけ見ると分かりやすいです。

    Q: ステータスコードはどれを覚えればいい？ A: まずは 200（成功）、404（見つからない）、500（サーバーエラー）の3つを覚えましょう。

    Q: GETとPOSTの違いが分からない A: URLを見てください。GETは情報がURL（アドレスバー）に表示されます。POSTは表示されません。

    重要なポイント

    開発者ツールは「透視メガネ」

    • 普段見えないWeb通信の詳細を確認できる
    • エラーの原因を特定するのに必須
    • 怖がらずに積極的に使うことが重要

    Networkタブは「通信記録簿」

    • Status列: 成功/失敗の結果
    • Method: GET/POSTの種類
    • Time列: 遅い原因の特定
    • Size列: データ量の確認

    ステータスコードは「結果報告書」

    • 2xx（緑）: うまくいった
    • 3xx（黄）: 他の場所を見て
    • 4xx（赤）: あなたの要求に問題
    • 5xx（赤）: サーバー側に問題

    GET/POSTは見た目でも分かる

    • GET: URLに情報が表示される
    • POST: URLに秘密情報は表示されない
    • 開発者ツールで詳細な通信内容を確認可能
