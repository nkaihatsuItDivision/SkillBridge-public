    🌿 1. ブランチとは何か？

    現実世界の例で理解する

    ブランチ（branch）とは、文字通り「枝」という意味です。プログラム開発では、メインの開発ラインから分岐して、並行して作業する仕組みのことです。

    📘 身近な例：雑誌編集部での記事制作

    想像してみてください。あなたが雑誌編集部で働いているとします：

    メイン記事（main）
    ↓
    ├── 特集記事A（feature/special-article-a）← 編集者Aが担当
    ├── インタビュー記事B（feature/interview-b）← 編集者Bが担当  
    └── コラム記事C（feature/column-c）← 編集者Cが担当
    • メイン記事: 確定した記事（安全・安定）
    • 各ブランチ: 各編集者が独立して作業
    • 完成後: メイン記事に統合（merge）

    プログラム開発でのブランチ

    同じように、プログラム開発でも：

    mainブランチ（安定版）
    ↓
    ├── feature/fix-deletion-bug（バグ修正）
    ├── feature/add-csv-export（CSV機能追加）
    └── feature/improve-ui（UI改善）

    ブランチを使う理由

    ❌ ブランチを使わない場合（危険）

    メインコード
    ↓ （直接編集）
プログラムが壊れる！😱
    ↓
他の人も作業できない...

    ✅ ブランチを使う場合（安全）

    mainブランチ（安全）
    ↓
feature/new-function ← ここで安全に開発
    ↓ （完成後）
mainブランチに統合 ← 安全にアップデート！

    異なるブランチ間を自由に移動してみましょう：

    mainブランチに戻る

    git switch main

    実行結果： 

    Switched to branch &apos;main&apos;

    機能ブランチに切り替え

    git switch feature/issue-1-fix-deletion-confirmation

    実行結果： 

    Switched to branch &apos;feature/issue-1-fix-deletion-confirmation&apos;

    重要な注意事項⚠️

    未コミットの変更がある場合

    ブランチ切り替え前に、必ず変更をコミットしてください：

    # 変更を確認
git status

# 変更をコミット
git add .
git commit -m &quot;Issue #1対応: 削除確認機能の改修作業中&quot;

    ブランチ切り替えは、クリーンな状態で行うのが安全です！

    // 削除確認（改修前）
private static boolean confirmDeletion(String studentName) {

}

    改修後（改善されたコード）

    // 削除確認（改修後）
private static boolean confirmDeletion(String studentName) {
    Scanner scanner = new Scanner(System.in);
    System.out.print(&quot;本当に &quot; + studentName + &quot; を削除しますか？ (y/n): &quot;);
    String input = scanner.next().toLowerCase(); // 小文字に統一

    // より多くの入力パターンに対応
    return input.equals(&quot;y&quot;) || input.equals(&quot;yes&quot;) ||

}

    🔀 6. git merge基本操作

    Feature Branch Workflowの実践

    Feature Branch Workflowは、企業開発で最も一般的なブランチ戦略です：

    1. mainブランチ: 常に安定した状態を保つ
    2. featureブランチ: 各機能を独立して開発
    3. merge: 機能完成後にmainブランチに統合

    mainブランチに戻って準備

    git switch main

    ブランチの状況確認

    git log --oneline --graph --all

    実行結果： 

    * 3a8f2b1 (feature/issue-1-fix-deletion-confirmation) Fix #1: 削除確認機能の入力判定を改善
* 2f4e8d9 (HEAD -&gt; main, origin/main) Add MIT License and improve README with badges
* 1a3c5e7 Add comprehensive Issue management system
* 9b2d4f6 Initial commit: Week 5 OOP student management system

    Fast-forward マージの実行

    Issue #1の修正をmainブランチに統合します：

    git merge feature/issue-1-fix-deletion-confirmation

    実行結果： 

    Updating 2f4e8d9..3a8f2b1
Fast-forward
 StudentCrudSystem.java | 5 ++++-
 1 file changed, 4 insertions(+), 1 deletion(-)

    Fast-forwardマージとは？

    Fast-forwardは、競合がない場合の単純なマージ方式です：

    Before:
main:     A --- B
feature:       \ --- C

After (Fast-forward):
main:     A --- B --- C

    mainブランチが単純に前進（fast-forward）します。

    // CSV出力機能の追加
private static void exportToCsv() {
    System.out.println(&quot;\n=== CSV出力機能 ===&quot;);

    if (students.isEmpty()) {
        System.out.println(&quot;出力する学生データがありません。&quot;);
        return;
    }

    try {
        java.io.FileWriter writer = new java.io.FileWriter(&quot;students.csv&quot;);

        // CSVヘッダー
        writer.write(&quot;名前,年齢,GPA,専攻\n&quot;);

        // 学生データ
        for (Student student : students) {
            writer.write(String.format(&quot;%s,%d,%.2f,%s\n&quot;,
                student.getName(),
                student.getAge(), 
                student.getGpa(),
                student.getMajor()));
        }

        writer.close();
        System.out.println(&quot;students.csvに出力完了しました！&quot;);

    } catch (java.io.IOException e) {
        System.out.println(&quot;CSV出力でエラーが発生しました: &quot; + e.getMessage());
    }
}

    ⚡ 8. 3-way mergeとコンフリクト解決

    意図的なコンフリクトの作成

    ## 🌿 ブランチ戦略

このプロジェクトでは**Feature Branch Workflow**を採用しています。

### ブランチ構成

- **`main`**: 安定版ブランチ。常にデプロイ可能な状態を維持
- **`feature/*`**: 機能開発ブランチ。Issue番号と機能名で命名
  - 例: `feature/issue-1-fix-deletion-confirmation`
  - 例: `feature/issue-2-add-csv-export`

### 開発フロー

1. **Issue作成**: GitHub Issuesで作業内容を明確化
2. **ブランチ作成**: `git switch -c feature/issue-X-description`
3. **機能開発**: 独立した環境で安全に開発
4. **コミット**: 明確なコミットメッセージで記録
5. **マージ**: mainブランチに統合後、フィーチャーブランチを削除

### コミットメッセージ規約

- **Fix #X**: バグ修正
- **Add #X**: 新機能追加
- **Update #X**: 既存機能改善
- **Remove #X**: 機能削除

    📝 重要なGit/GitHub用語集

    ブランチ関連

    • Branch（ブランチ）: 開発ラインの分岐
    • Feature Branch: 機能開発専用ブランチ
    • Main Branch: メインの安定ブランチ
    • HEAD: 現在作業中のブランチのポインタ

    マージ関連

    • Merge（マージ）: ブランチの統合
    • Fast-forward: 単純な前進統合
    • 3-way merge: 複数変更点の統合
    • Conflict（コンフリクト）: 統合時の競合

    Git操作

    • Switch: ブランチ切り替え（新コマンド）
    • Checkout: ブランチ切り替え（旧コマンド）
    • Origin: デフォルトのリモートリポジトリ名
    • Upstream: 上流リポジトリ（元のリポジトリ）
