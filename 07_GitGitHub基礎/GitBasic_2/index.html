    1. Gitとは何か：分散バージョン管理システムの理解

    Gitの基本的な定義

    Gitは、プログラムのソースコード等の変更履歴を記録・追跡するための分散型バージョン管理システムです。

    📘 Gitの概念概要

    従来のバージョン管理との違い

    ❌ 手作業でのバージョン管理の問題

    プロジェクトフォルダ/
├── StudentManager_v1.java
├── StudentManager_v2.java
├── StudentManager_v2_修正版.java
├── StudentManager_最新.java
├── StudentManager_最新_本当に最新.java
└── StudentManager_最終版.java  ← 結局どれが最新？

    ✅ Gitでの管理

    プロジェクトフォルダ/
├── .git/（Git管理情報）
└── src/
    ├── Person.java
    ├── Student.java
    └── StudentManager.java  ← 1つのファイルで全履歴管理

    Gitの核心的な特徴

    1. 分散型管理

    • 各開発者が完全な履歴を持つ
    • 中央サーバーが停止しても作業継続可能
    • オフライン環境でも履歴管理が可能

    2. 変更の追跡

    • ファイル単位ではなく変更内容を記録
    • どの行が変更されたかを正確に把握
    • 変更理由や内容をメッセージで記録

    3. ブランチ機能

    • 複数の開発ラインを並行管理
    • 実験的な機能開発を安全に実施
    • 完成した機能のみを統合可能

    2. リポジトリ：プロジェクトの「データベース」

    リポジトリの基本概念

    リポジトリ（Repository）は、プロジェクトのすべてのファイルと変更履歴を保存するデータベースのような仕組みです。

    📘 リポジトリ構造の図解

    Week 5システムでのリポジトリイメージ

    学生管理システム リポジトリ
├── 📁 プロジェクトファイル
│   ├── Person.java
│   ├── Student.java
│   ├── GraduateStudent.java
│   ├── UndergraduateStudent.java
│   ├── StudentManager.java
│   └── （その他のファイル）
├── 📜 変更履歴
│   ├── 2024/12/01 - 基本クラス作成
│   ├── 2024/12/02 - 継承機能追加
│   ├── 2024/12/03 - インターフェース実装
│   ├── 2024/12/04 - 例外処理追加
│   └── 2024/12/05 - Lombok統合
└── 🏷️ バージョン情報
    ├── v1.0 - 基本機能完成
    ├── v2.0 - OOP機能統合
    └── v3.0 - エンタープライズ版完成

    リポジトリの種類

    ローカルリポジトリ

    • 自分のPC内にあるリポジトリ
    • 個人作業時のメイン作業場
    • オフラインでも履歴管理可能

    リモートリポジトリ

    • GitHub等のサーバー上にあるリポジトリ
    • チーム共有・バックアップ・公開用
    • 複数人での協業の中心拠点

    3. コミット：変更の「セーブポイント」

    コミットの基本概念

    コミット（Commit）は、プロジェクトの変更内容を履歴として保存する「セーブポイント」の作成です。

    📘 コミットによる保存ポイントの概念

    Week 5システム開発でのコミット例

    コミット1: 基本クラス作成

    変更内容：
├── Person.java（新規作成）
└── Student.java（新規作成）

コミットメッセージ：
「基本的なPersonクラスとStudentクラスを作成」

保存される情報：
- 変更したファイル
- 変更した内容（追加・削除・修正）
- 変更した日時
- 変更者の情報
- 変更理由（メッセージ）

    コミット2: 継承機能追加

    変更内容：
├── Person.java（修正）
├── Student.java（修正：継承関係追加）
├── GraduateStudent.java（新規作成）
└── UndergraduateStudent.java（新規作成）

コミットメッセージ：
「継承機能実装：Person→Student階層構造完成」

    コミット3: Lombok統合

    変更内容：
├── Person.java（修正：@Data追加）
├── Student.java（修正：@Builder追加）
├── GraduateStudent.java（修正：Lombok化）
└── UndergraduateStudent.java（修正：Lombok化）

コミットメッセージ：
「Lombok統合で86%コード削減を実現」

    コミットの重要な特徴

    1. 原子性（Atomicity）

    • 1つのコミット = 1つの論理的な変更
    • 関連する変更を1つのまとまりとして記録
    • 「部分的に適用」されることがない

    2. 一意性（Uniqueness）

    • 各コミットには固有のID（ハッシュ値）が付与
    • 同じ変更内容でも異なるコミットとして識別
    • 履歴の中で確実に特定可能

    3. 不変性（Immutability）

    • 一度作成されたコミットは変更されない
    • 過去の状態を確実に保持
    • 改変されていないことを保証

    コミットメッセージの重要性

    ✅ 良いコミットメッセージの例

    ✅ 「新規登録機能のバリデーション強化」
✅ 「学生検索機能の性能改善（O(n)→O(log n)）」
✅ 「例外処理追加：不正な学生IDの対応」
✅ 「Lombok導入によるボイラープレートコード削減」

    ❌ 悪いコミットメッセージの例

    ❌ 「修正」
❌ 「バグ修正」
❌ 「更新」
❌ 「テスト」

    4. 履歴管理：タイムライン的な変更追跡

    履歴管理の基本概念

    Gitの履歴管理は、プロジェクトの成長過程を時系列で追跡する仕組みです。

    📘 プロジェクト成長履歴の概念

    Week 5システムの開発履歴例

    📅 Week 5システム開発タイムライン

12月1日  🎯 基本設計完成
commit a1b2c3d4: "Person・Studentクラス基本構造完成"

12月2日  🔄 継承機能実装
commit e5f6g7h8: "継承関係実装：Person→Student階層"

12月3日  🎨 ポリモーフィズム追加
commit i9j0k1l2: "学部生・大学院生クラス分離実装"

12月4日  🛡️ 例外処理強化
commit m3n4o5p6: "カスタム例外4種実装：エラー処理完備"

12月5日  ⚡ Lombok統合
commit q7r8s9t0: "Lombok導入：86%コード削減達成"

12月6日  🏆 システム完成
commit u1v2w3x4: "エンタープライズ版学生管理システム完成"

    履歴から得られる重要な情報

    1. 変更の経緯

    • いつ その機能が追加されたか
    • なぜ その変更が必要だったか
    • どのように 実装されたか
    • 誰が 変更を行ったか

    2. プロジェクトの成長過程

    • 機能の追加順序
    • 設計思想の変化
    • 技術的な進歩
    • 品質向上の軌跡

    3. 問題解決の手がかり

    • バグが発生した時期の特定
    • 機能退行の原因調査
    • 成功した実装パターンの参照
    • リファクタリングの効果測定

    履歴の活用例：バグ発生時の対応

    問題発生：
「学生の成績計算が正しく動作しない」

履歴を確認：
commit m3n4o5p6: "カスタム例外4種実装：エラー処理完備"
├── この変更後から問題が発生している
├── 変更内容を確認
└── 問題のある箇所を特定

対応：
1. 該当コミットの変更内容を詳細確認
2. 問題のある部分を修正
3. 修正内容を新しいコミットとして記録

    📘 コミット履歴のタイムライン

    5. 分散バージョン管理の理解

    中央集権型 vs 分散型

    従来の中央集権型（SVN等）

    中央サーバー
    ↓
  全履歴
    ↓
開発者A ← 部分的な情報
開発者B ← 部分的な情報
開発者C ← 部分的な情報

    問題点： - サーバー停止時は作業不可 - 履歴情報が分散されていない - オフライン作業の制限

    Gitの分散型

    開発者A     開発者B     開発者C
完全な履歴   完全な履歴   完全な履歴
    ↓          ↓          ↓
      GitHub（共有拠点）
        完全な履歴

    メリット： - 各自が完全な履歴を保持 - オフライン作業が可能 - 単一障害点が存在しない

    📘 分散型と集中型の比較

    Week 5システムでの分散開発イメージ

    田中さん（フロントエンド担当）

↓ GitHub で統合 ↓

└── テスト機能

    6. Gitのワークフロー概念

    Gitの3つのエリア

    📘 Gitの3つの作業エリアのワークフロー

    ワーキングディレクトリ

    • 実際にファイルを編集する場所
    • 現在作業中のファイル状態
    • Week 5システムのファイルがある場所

    ステージングエリア

    • コミット予定の変更を準備する場所
    • 「次のコミットに含める変更」を選択
    • 部分的な変更の選択が可能

    リポジトリ

    • コミットされた履歴が保存される場所
    • プロジェクトの確定した履歴
    • 過去のバージョンへのアクセス拠点

    変更のライフサイクル

    1. ファイル編集
   Student.java を修正
   ↓
2. ステージング
   修正内容をコミット準備エリアに追加
   ↓
3. コミット
   変更をリポジトリの履歴として確定
   ↓
4. プッシュ（後のレッスンで学習）
   リモートリポジトリに共有

    💡 よくある質問

    Q1: コミットはどの程度の頻度で行うべきですか？

    A: 以下の基準を参考にしてください：

    適切なコミット頻度

    • 機能完成時: 1つの機能が動作するようになった時
    • 論理的な区切り: 意味のある変更のまとまりごと
    • 1日1回以上: 作業終了時には必ずコミット

    Week 5システム開発での例

    // 悪い例：すべてを1つのコミットに
commit: "学生管理システム完成" 
└── 全ファイル600行一括変更

// 良い例：段階的なコミット
commit: "Person基底クラス作成"
commit: "Student継承クラス追加" 
commit: "例外処理機能実装"
commit: "Lombok統合による最適化"

    Q2: コミットメッセージはどう書けばいいですか？

    A: 以下のパターンを参考にしてください：

    効果的なメッセージパターン

    ✅ 「[種別] 概要：詳細」形式
[機能追加] 学生検索：名前での部分一致検索機能
[バグ修正] 成績計算：平均値算出時の小数点処理
[改善] 性能：学生一覧表示の処理速度向上
[リファクタ] コード整理：重複処理の共通化

    Q3: ファイルを削除した場合はどうなりますか？

    A: Gitでは削除も「変更」として記録されます：

    // ファイル削除もコミットで記録
commit: "不要なテストファイル削除"
└── TestData.java 削除

// 必要に応じて過去のコミットから復元可能
git checkout コミットID -- TestData.java

    Q4: 間違ってコミットした場合は修正できますか？

    A: 状況に応じて修正方法があります：

    直前のコミットの場合

    # コミットメッセージのみ修正
git commit --amend -m "正しいメッセージ"

# ファイルの追加も可能
git add 忘れたファイル.java
git commit --amend

    過去のコミットの場合

    • 新しいコミットで修正内容を追加
    • 履歴の改変は推奨されない（特にチーム開発）

    Q5: バイナリファイル（画像・音声等）もGitで管理できますか？

    A: 管理可能ですが注意が必要です：

    管理可能なバイナリファイル

    ✅ 小さな画像ファイル（アイコン等）
✅ 設定ファイル（小サイズ）
✅ ドキュメント（少数）

    避けるべきファイル

    ❌ 大きな動画ファイル
❌ 頻繁に変更される一時ファイル
❌ ビルド成果物（.class、.jar等）

    🎯 まとめ

    このレッスンで学んだGitの基本概念：

    技術的理解

    • Git: 分散型バージョン管理システムの特徴
    • リポジトリ: プロジェクトの完全なデータベース
    • コミット: 変更の確定セーブポイント作成
    • 履歴管理: 時系列での変更追跡システム
    • 分散型: 各開発者が完全な履歴を保持

    実用的理解

    • Week 5システム: 600-700行システムでの具体的活用イメージ
    • 開発フロー: 編集→ステージング→コミット→共有
    • チーム開発: 複数人での安全な協業方法
    • 問題解決: 履歴を活用したバグ特定・修正
    • 品質管理: 変更理由の記録と追跡

    概念的理解

    • 変更の追跡: ファイル単位ではなく変更内容の管理
    • 時間軸管理: プロジェクトの成長過程の記録
    • 安全性: 過去バージョンへの確実な復元能力
    • 協業性: 複数人でのコンフリクトのない開発
    • 透明性: すべての変更の可視化と共有
