    1. push・pullとは何か？

    Git操作の全体像

    📘 GitのPush/Pullの概要

    これまでのGit操作はすべてローカルで完結していました。今回学ぶのは、ローカルとリモートの同期です。

    基本概念の理解

    git push（プッシュ）

    • 方向: ローカル → リモート（GitHub）
    • 目的: 自分の変更を世界に公開
    • タイミング: コミット後、他の人に共有したいとき

    git pull（プル）

    • 方向: リモート（GitHub） → ローカル
    • 目的: 他の人の変更を自分の環境に取得
    • タイミング: 作業開始時、最新状態にしたいとき

    💡 現実世界の例え

    図書館での協同作業に例えると：

    • push: 自分が書いた原稿を図書館に納品する
    • pull: 図書館から最新版の資料を借りてくる

    複数の人が同じ本を編集する場合、みんなが最新版を共有する必要がありますよね？

## 📝 プロジェクト概要

Java のオブジェクト指向プログラミング（OOP）の応用技術を活用して開発した、実用的な学生管理システムです。

- **Lombok**: コード生成によるボイラープレート削減（86%削減）

    7. push・pullの理解確認

    🧠 概念整理

    今回の体験を通じて理解した概念を整理しましょう：

    git push（ローカル → リモート）

    [ローカル] git commit → git push → [GitHub]
           コミット      送信       公開

    git pull（リモート → ローカル）

    [GitHub] Web編集/他者変更 → git pull → [ローカル]
         新コミット         取得      反映

    データの流れ図

    📘 Gitデータフローの概要

    実際の協業フロー

    実際のチーム開発では、以下のサイクルが繰り返されます：

    1. 作業開始時: git pull で最新状態にする
    2. ローカル開発: コード修正・テスト・コミット
    3. 共有時: git push で変更を公開
    4. 他者の変更取得: 定期的に git pull で同期

    💡 重要なルール

    push前のpull習慣

    # 推奨：pushする前に必ずpull
git pull
git push

    頻繁な同期

    • 朝の開始時：git pull
    • 他者の変更時：git pull
    • 自分の変更完了時：git push

    9. よくある質問とトラブルシューティング

    Q1: pushが失敗する場合

    エラー例： 

    ! [rejected]        main -&gt; main (non-fast-forward)
error: failed to push some refs to &apos;github.com:username/repository.git&apos;

    原因: リモートに新しい変更があり、ローカルが古い

    解決方法： 

    git pull    # まず最新状態に更新
git push    # その後プッシュ

    Q2: pullで競合（conflict）が発生

    エラー例： 

    Auto-merging README.md
CONFLICT (content): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.

    対処法： 1. 競合ファイルを手動編集 2. git add 競合ファイル名 3. git commit -m &quot;競合解決&quot;

    Q3: 間違ったファイルをpushしてしまった

    対処法： 

    # ファイルを削除してコミット
git rm 間違ったファイル名
git commit -m &quot;不要ファイル削除&quot;
git push

    Q4: コミットメッセージを間違えた

    直前のコミットの場合： 

    git commit --amend -m &quot;正しいメッセージ&quot;
git push --force    # 注意：強制プッシュ

    ⚠️ 注意: --force は慎重に使用してください。

    Q5: リモートURLを確認・変更したい

    確認： 

    git remote -v

    変更： 

    git remote set-url origin 新しいURL

    10. セキュリティとベストプラクティス

    🔒 セキュリティ注意事項

    絶対にpushしてはいけないもの

    • パスワード・APIキー
    • 秘密鍵・認証情報
    • 個人情報・機密情報
    • 大容量ファイル（100MB以上）

    適切な.gitignore設定

    プロジェクトルートに .gitignore ファイルを作成：

    notepad .gitignore

    内容： 

    # IDE設定ファイル
.metadata/
.settings/
.project
.classpath

# 一時ファイル
*.tmp
*.log
*~

# 機密情報
config/secrets.properties
*.key
*.pem

# 大容量ファイル
*.zip
*.rar

    🏆 ベストプラクティス

    コミットメッセージの書き方

    # ✅ 良い例
git commit -m &quot;READMEに使用方法セクション追加&quot;
git commit -m &quot;バグ修正：null値チェック追加&quot;
git commit -m &quot;機能追加：CSVエクスポート機能実装&quot;

# ❌ 悪い例
git commit -m &quot;修正&quot;
git commit -m &quot;update&quot;
git commit -m &quot;aaa&quot;

    プッシュの頻度

    • 小さな変更: 複数まとめてプッシュ
    • 重要な機能: 完成したらすぐプッシュ
    • 作業終了時: 必ずプッシュして保護

    プル習慣

    • 作業開始時: 必ず git pull
    • プッシュ前: 必ず git pull
    • 定期的: 1日数回の同期
