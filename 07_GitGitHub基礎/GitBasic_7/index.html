    1. push・pullとは何か？

    Git操作の全体像

    📘 GitのPush/Pullの概要

    これまでのGit操作はすべてローカルで完結していました。今回学ぶのは、ローカルとリモートの同期です。

    基本概念の理解

    git push（プッシュ）

    • 方向: ローカル → リモート（GitHub）
    • 目的: 自分の変更を世界に公開
    • タイミング: コミット後、他の人に共有したいとき

    git pull（プル）

    • 方向: リモート（GitHub） → ローカル
    • 目的: 他の人の変更を自分の環境に取得
    • タイミング: 作業開始時、最新状態にしたいとき

    7. push・pullの理解確認

    🧠 概念整理

    今回の体験を通じて理解した概念を整理しましょう：

    git push（ローカル → リモート）

    [ローカル] git commit → git push → [GitHub]
           コミット      送信       公開

    git pull（リモート → ローカル）

    [GitHub] Web編集/他者変更 → git pull → [ローカル]
         新コミット         取得      反映

    データの流れ図

    📘 Gitデータフローの概要

    実際の協業フロー

    実際のチーム開発では、以下のサイクルが繰り返されます：

    1. 作業開始時: git pull で最新状態にする
    2. ローカル開発: コード修正・テスト・コミット
    3. 共有時: git push で変更を公開
    4. 他者の変更取得: 定期的に git pull で同期

    💡 重要なルール

    push前のpull習慣

    # 推奨：pushする前に必ずpull
git pull
git push

    頻繁な同期

    • 朝の開始時：git pull
    • 他者の変更時：git pull
    • 自分の変更完了時：git push

    9. よくある質問とトラブルシューティング

    Q1: pushが失敗する場合

    エラー例： 

    ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'github.com:username/repository.git'

    原因: リモートに新しい変更があり、ローカルが古い

    解決方法： 

    git pull    # まず最新状態に更新
git push    # その後プッシュ

    Q2: pullで競合（conflict）が発生

    エラー例： 

    Auto-merging README.md
CONFLICT (content): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.

    対処法： 1. 競合ファイルを手動編集 2. git add 競合ファイル名 3. git commit -m "競合解決"

    Q3: 間違ったファイルをpushしてしまった

    対処法： 

    # ファイルを削除してコミット
git rm 間違ったファイル名
git commit -m "不要ファイル削除"
git push

    Q4: コミットメッセージを間違えた

    直前のコミットの場合： 

    git commit --amend -m "正しいメッセージ"
git push --force    # 注意：強制プッシュ

    ⚠️ 注意: --force は慎重に使用してください。

    Q5: リモートURLを確認・変更したい

    確認： 

    git remote -v

    変更： 

    git remote set-url origin 新しいURL
