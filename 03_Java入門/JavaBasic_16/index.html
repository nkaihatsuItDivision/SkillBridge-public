    1. なぜ「設計」が必要なのか？

    🤔 素朴な疑問

    「レッスン15では、特に設計書を作らずにプログラムを完成させました。なぜわざわざ設計が必要なのでしょうか？」

    この疑問は、プログラミング初心者の誰もが感じることです。

    🏗️ 建物の建設で考えてみよう

    設計図なしで家を建てる場合： - 部屋の大きさがバラバラ - 配線や配管の位置が適当 - 後から変更が困難 - 全体の統一感がない - 完成まで時間がかかる

    設計図ありで家を建てる場合： - 事前に全体像が見える - 材料や工期が計算できる - 問題を事前に発見・解決 - 複数人での作業が可能 - 品質が安定する

    💻 プログラム開発でも同じ

    設計なしの開発： - 行き当たりばったりの実装 - 後から機能追加が困難 - バグが発生しやすい - 他の人には理解しにくい - 修正に時間がかかる

    設計ありの開発： - 全体の構造が明確 - 段階的な実装が可能 - バグを事前に防げる - チーム開発が可能 - 保守・拡張が容易

    📘 建築とプログラミングの比較

    2. ソフトウェア開発ライフサイクル（SDLC）

    📋 開発の標準的な流れ

    プロの開発現場では、以下の5段階で開発を進めます：

    1. 要件定義 → 2. 設計 → 3. 実装 → 4. テスト → 5. 運用・保守

    各段階の役割

    1. 要件定義（何を作るか）

    • 目的: 作るシステムの機能を明確にする
    • 成果物: 要件定義書、機能一覧
    • 質問: 「何ができれば良いのか？」

    2. 設計（どのように作るか）

    • 目的: システムの構造を決める
    • 成果物: 設計書、クラス図、メソッド一覧
    • 質問: 「どのような構造にするか？」

    3. 実装（実際に作る）

    • 目的: 設計に基づいてコードを書く
    • 成果物: プログラムファイル
    • 質問: 「どのようにコードを書くか？」

    4. テスト（正しく動くか）

    • 目的: バグを発見・修正する
    • 成果物: テスト結果、修正版
    • 質問: 「仕様通りに動作するか？」

    5. 運用・保守（使い続ける）

    • 目的: システムを維持・改善する
    • 成果物: 運用マニュアル、改善版
    • 質問: 「問題なく使い続けられるか？」

    要件が決まったら、次はどのような構造でシステムを作るかを決めます。

    設計の3つの視点

    1. データ設計（何を記録するか）

    // 学生情報をどのように記録するか
String[] names = new String[MAX_STUDENTS];
int[] scores = new int[MAX_STUDENTS];
int count = 0;  // 現在の学生数

    2. 機能設計（どのような処理をするか）

    // 必要なメソッドの洗い出し
- showMenu()           // メニュー表示
- addStudent()         // 学生追加
- showStudents()       // 学生一覧表示
- deleteStudent()      // 学生削除
- showStatistics()     // 統計表示

    3. 流れ設計（どのような順序で処理するか）

    // プログラム全体の流れ
1. システム開始
2. メニュー表示
3. ユーザー選択
4. 選択に応じた処理実行
5. 結果表示
6. メニューに戻る（または終了）

    6. テスト・デバッグの基本

    🔍 テストの重要性

    プログラムは「書いたら終わり」ではありません。正しく動作するかを確認する必要があります。

    テストの種類

    1. 単体テスト（ユニットテスト）

    • 目的: 各メソッドが個別に正しく動作するか
    • 例: addStudent()メソッドが正しく学生を追加するか

    2. 統合テスト

    • 目的: メソッド同士の連携が正しく動作するか
    • 例: 学生追加→表示の流れが正しく動作するか

    3. システムテスト

    • 目的: システム全体が要件通りに動作するか
    • 例: ユーザーシナリオ通りに操作できるか

    テスト計画の例

    【学生管理システム テスト計画】

■ 正常系テスト
T001: 学生を1名登録し、正しく表示されることを確認
T002: 複数の学生を登録し、一覧表示が正しいことを確認
T003: 学生を削除し、リストから消えることを確認
T004: 統計機能で正しい平均点が表示されることを確認

■ 異常系テスト
T005: 不正な点数（負の数、101以上）を入力した場合の動作確認
T006: 存在しない番号で削除を試みた場合の動作確認
T007: 最大人数を超えて登録を試みた場合の動作確認
T008: 学生が0名の時の統計表示の動作確認

■ 境界値テスト
T009: 点数0点、100点での動作確認
T010: 最大人数ちょうどでの登録確認
T011: 名前が空文字の場合の動作確認

    🛠️ Eclipse デバッガーの基本使用法

    デバッガーとは？

    プログラムの実行を一時停止し、変数の値や実行の流れを詳しく調べるツールです。

    基本的な使い方

    1. ブレークポイントの設定 

    public static void addStudent() {
    System.out.print("学生の名前を入力してください: ");
    String name = scanner.nextLine();  // ← この行の左側をクリック
    // ...
}
    行番号の左側をクリックすると、青い丸（ブレークポイント）が表示されます。

    2. デバッグ実行 - 通常の実行ボタンの横にある「虫」のアイコンをクリック - ブレークポイントで実行が停止します

    3. ステップ実行 - F6: 次の行に進む（ステップオーバー） - F5: メソッドの中に入る（ステップイン） - F8: 次のブレークポイントまで実行（Resume）

    4. 変数の確認 - デバッグ中は「Variables」ビューで変数の値を確認できます

    デバッグの活用例

    public static void addStudent() {
    if (count >= MAX_STUDENTS) {
        System.out.println("これ以上登録できません。");
        return;
    }

    System.out.print("学生の名前を入力してください: ");
    String name = scanner.nextLine();  // ← ブレークポイント

    // この時点でnameの値を確認
    // countの値も確認

    System.out.print("点数を入力してください (0-100): ");
    int score = getValidScore();  // ← ここでも確認

    names[count] = name;    // ← 配列に正しく格納されているか確認
    scores[count] = score;
    count++;

    System.out.println("学生を追加しました: " + name + " (" + score + "点)");
}

    public static void main(String[] args) {
    System.out.println("本の在庫管理システム");

        try {

    public static void addBook() {
    if (bookCount >= MAX_BOOKS) {

        return;
    }

    System.out.print("本のタイトルを入力してください: ");

}

public static int getValidNumber(String prompt) {
    while (true) {
        try {
            int number = scanner.nextInt();
            scanner.nextLine(); // 改行文字消費

            if (number >= 0) {
                return number;
            } else {
                System.out.print("0以上の数字を入力してください: ");
            }
        } catch (Exception e) {
            System.out.print("数字を入力してください: ");
            scanner.nextLine(); // 不正入力を消費
        }
    }
}

    ステップ4: 一覧表示機能

    public static void showBooks() {
    if (bookCount == 0) {
        System.out.println("登録されている本がありません。");
        return;
    }

    System.out.println("\n=== 在庫一覧 ===");
    System.out.printf("%-3s %-20s %s%n", "No.", "タイトル", "在庫数");
    System.out.println("--------------------------------");

    for (int i = 0; i < bookCount; i++) {
        System.out.printf("%-3d %-20s %d冊%n", 
                         (i + 1), titles[i], stocks[i]);
    }
}

    ステップ5: 在庫更新機能

    public static void updateStock() {
    if (bookCount == 0) {
        System.out.println("登録されている本がありません。");
        return;
    }

    showBooks();
    System.out.print("更新する本の番号を入力してください: ");

    try {
        int bookIndex = scanner.nextInt() - 1;
        scanner.nextLine();

        if (bookIndex < 0 || bookIndex >= bookCount) {
            System.out.println("無効な番号です。");
            return;
        }

        System.out.println("選択された本: " + titles[bookIndex] + 
                          " (現在の在庫: " + stocks[bookIndex] + "冊)");

        System.out.print("新しい在庫数を入力してください: ");
        int newStock = getValidNumber("在庫数");

        stocks[bookIndex] = newStock;
        System.out.println("在庫数を更新しました: " + titles[bookIndex] + 
                          " (在庫: " + newStock + "冊)");

    } catch (Exception e) {
        System.out.println("数字を入力してください。");
        scanner.nextLine();
    }
}

    8. よくある質問

    Q1: 設計に時間をかけすぎて、実装時間が足りなくなってしまいます

    A: 最初は簡単な設計から始めましょう。 - 必要なデータを整理する（5分） - 必要なメソッドをリストアップする（5分） - 大まかな処理の流れを考える（5分）

    完璧な設計を目指さず、「まあまあの設計」で実装を始めることが重要です。

    Q2: 設計通りに実装できない場合はどうしますか？

    A: 設計を修正してください！設計は絶対ではありません。 実装中に「もっと良い方法」が見つかることはよくあります。 - 設計は「たたき台」として考える - 実装しながら改善していく - 大きな変更の場合は一度立ち止まって再設計する

    Q3: テストはどこまでやれば良いですか？

    A: 最低限、以下の確認をしましょう： - 正常系: 想定通りの入力で正しく動作するか - 異常系: 不正な入力でもエラーにならないか - 境界値: 最大値、最小値で正しく動作するか

    完璧を目指さず、「多分大丈夫」から「確実に大丈夫」のレベルに上げることが目標です。

    Q4: デバッガーを使うタイミングがわかりません

    A: 以下の状況でデバッガーを使ってみてください： - バグが発生したとき → どこで問題が起きているか調査 - 期待と違う結果になったとき → 変数の値を確認 - 複雑な処理を理解したいとき → 一行ずつ実行して流れを確認

    最初は「なんとなく使ってみる」程度で十分です。
