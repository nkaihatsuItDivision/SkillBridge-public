    学習目標

    • クラスとオブジェクトの概念を理解する
    • クラスの定義方法とインスタンス化を学ぶ
    • フィールド、プロパティ、メソッドを使ったクラス設計ができる
    • オブジェクト指向プログラミングの利点を理解する

    クラスとオブジェクトとは

    クラスとオブジェクトの概念

    • クラス: オブジェクトの設計図（データと処理をまとめた型）
    • オブジェクト: クラスから作られた実際のもの（インスタンス）

    身近な例で理解する

    • たい焼きの型（クラス）と実際のたい焼き（オブジェクト）
    • 型は1つだが、そこから何個でもたい焼きを作れる

    • 各たい焼きは独立している（1つを食べても他に影響しない）

    • アプリの設計書（クラス）とスマホにインストールされたアプリ（オブジェクト）

    • 同じアプリでも、各スマホで独立して動作する
    • 設定や データは端末ごとに異なる

    プログラムでの例

    // Student クラス = 「学生」の設計図
public class Student { ... }

// 実際の学生オブジェクト = クラスから作られた個別の学生
Student tanaka = new Student();  // 田中さんという学生
Student sato = new Student();    // 佐藤さんという学生
// 田中さんと佐藤さんは別々の存在（独立している）

    なぜ必要なのか

    これまでは「学生1」「学生2」という変数を個別に管理していましたが、 クラスを使うことで「学生」という概念をプログラムで表現できるようになります。

    アクセス修飾子とは

    クラスの外部から、そのメンバー（フィールド、プロパティ、メソッド）に アクセスできるかを制御する仕組みです。

    public（公開）

    クラスの外部からアクセス可能

    public class Student
{
    public string Name;  // 外部から読み書き可能

    public void Study()  // 外部から呼び出し可能
    {
        Console.WriteLine(Name + &quot;が勉強しています&quot;);
    }
}

Student student = new Student();
student.Name = &quot;田中&quot;;  // OK
student.Study();       // OK

    private（非公開）

    public class Student
{
    private int secretScore;  // 外部からアクセス不可

    private void CalculateGrade()  // 外部から呼び出し不可
    {
        // クラス内部からのみ使用
    }
}

Student student = new Student();
student.secretScore = 100;    // エラー！
student.CalculateGrade();     // エラー！

    なぜprivateが必要か

    • データ保護：重要なデータを外部から勝手に変更されるのを防ぐ
    • 内部実装の隠蔽：クラス内部でのみ使う処理を隠す
    • 安全性向上：予期しない操作を防ぐ

    基本的なクラスの定義

    public class Student
{
    // フィールド（データ）
    public string Name;
    public int Age;
    public int Score;

    // メソッド（処理）
    public void DisplayInfo()
    {
        Console.WriteLine(&quot;名前: &quot; + Name + &quot;, 年齢: &quot; + Age + &quot;, 点数: &quot; + Score);
    }

    public string GetGrade()
    {
        if (Score &gt;= 90) return &quot;A&quot;;
        if (Score &gt;= 80) return &quot;B&quot;;
        if (Score &gt;= 70) return &quot;C&quot;;
        if (Score &gt;= 60) return &quot;D&quot;;
        return &quot;F&quot;;
    }
}

    オブジェクトの作成と使用

    using System;

namespace ClassExample
{
    class Program
    {
        static void Main(string[] args)
        {
            // オブジェクトの作成（インスタンス化）
            Student student1 = new Student();

            // フィールドに値を設定
            student1.Name = &quot;田中太郎&quot;;
            student1.Age = 20;
            student1.Score = 85;

            // メソッドの呼び出し
            student1.DisplayInfo();
            Console.WriteLine(&quot;評価: &quot; + student1.GetGrade());

            // 別のオブジェクトを作成
            Student student2 = new Student();
            student2.Name = &quot;佐藤花子&quot;;
            student2.Age = 19;
            student2.Score = 92;

            student2.DisplayInfo();
            Console.WriteLine(&quot;評価: &quot; + student2.GetGrade());
        }
    }
}

    コンストラクタ

    コンストラクタとは

    オブジェクトが作成される瞬間に自動実行される特別なメソッドです。 オブジェクトの「初期設定」を行うために使用します。

    なぜコンストラクタが必要なのか

    // コンストラクタなしの場合
Student student = new Student();
student.Name = &quot;田中&quot;;     // 設定し忘れの可能性
student.Age = 20;         // 設定し忘れの可能性
student.Score = 85;       // 設定し忘れの可能性
// コンストラクタありの場合
Student student = new Student(&quot;田中&quot;, 20, 85);  // 作成時に必ず設定

    コンストラクタの特徴

    • メソッド名がクラス名と同じ
    • 戻り値の型を書かない
    • new演算子でオブジェクト作成時に自動実行
    • 複数定義可能（オーバーロード）

    実行タイミング

    Console.WriteLine(&quot;1. オブジェクト作成前&quot;);
Student student = new Student(&quot;田中&quot;, 20);  // ← この瞬間にコンストラクタ実行
Console.WriteLine(&quot;3. オブジェクト作成後&quot;);

// コンストラクタ内での処理
public Student(string name, int age)
{
    Console.WriteLine(&quot;2. コンストラクタ実行中&quot;);
    Name = name;
    Age = age;
}

    デフォルトコンストラクタとパラメータ付きコンストラクタ

    public class Student
{
    public string Name;
    public int Age;
    public int Score;

    // デフォルトコンストラクタ
    public Student()
    {
        Name = &quot;未設定&quot;;
        Age = 0;
        Score = 0;
    }

    // パラメータ付きコンストラクタ
    public Student(string name, int age, int score)
    {
        Name = name;
        Age = age;
        Score = score;
    }

    public void DisplayInfo()
    {
        Console.WriteLine(&quot;名前: &quot; + Name + &quot;, 年齢: &quot; + Age + &quot;, 点数: &quot; + Score);
    }
}

    使用例

    // デフォルトコンストラクタを使用
Student student1 = new Student();

// パラメータ付きコンストラクタを使用
Student student2 = new Student(&quot;田中太郎&quot;, 20, 85);

    プロパティ

    プロパティとは

    フィールドへの安全なアクセスを提供する仕組みです。 「フィールドのようにアクセスできるメソッド」と考えてください。

    フィールドとプロパティの違い

    フィールドの場合（問題あり）

    public class Student
{
    public int Score;  // 直接アクセス可能
}

Student student = new Student();
student.Score = -50;   // 異常値でも設定されてしまう
student.Score = 150;   // 範囲外でも設定されてしまう

    プロパティの場合（安全）

    public class Student
{
    private int score;  // フィールドは非公開

    public int Score    // プロパティで安全なアクセス
    {
        get { return score; }
        set 
        { 
            if (value &gt;= 0 &amp;&amp; value &lt;= 100)
                score = value;
            else
                Console.WriteLine(&quot;点数は0-100で入力してください&quot;);
        }
    }
}

Student student = new Student();
student.Score = -50;   // エラーメッセージが表示され、設定されない

    getterとsetter

    • getter（get）：値を取得する処理
    • setter（set）：値を設定する処理
    • value：setterで設定しようとしている値

    自動プロパティとは

    プロパティの簡潔な書き方で、C#が内部で自動的にprivateフィールドを作成してくれる機能です。

    従来のプロパティ

    public class Student
{
    private string name;    // 手動でprivateフィールドを宣言

    public string Name      // プロパティを手動で定義
    {
        get { return name; }
        set { name = value; }
    }
}

    自動プロパティ（完結版）

    public class Student
{
    public string Name { get; set; }  // これだけでOK！

    // 内部で以下のコードが自動生成される：
    // private string &lt;Name&gt;k__BackingField;
    // public string Name 
    // {
    //     get { return &lt;Name&gt;k__BackingField; }
    //     set { &lt;Name&gt;k__BackingField = value; }
    // }
}

    自動プロパティが適している場面

    • 単純な値の保存・取得のみを行う場合
    • 特別な検証やロジックが不要な場合

    例

    public class Student
{
    public string Name { get; set; }        // 単純な文字列
    public string StudentId { get; set; }   // 単純なID
    public DateTime BirthDate { get; set; } // 単純な日付
}

    従来のプロパティが必要な場面

    • 値の検証が必要な場合
    • 特別な処理が必要な場合

    例

    public class Student
{
    private int age;

    public int Age  // 検証が必要なので従来のプロパティ
    {
        get { return age; }
        set 
        { 
            if (value &gt;= 0 &amp;&amp; value &lt;= 150)
                age = value;
            else
                Console.WriteLine(&quot;年齢は0-150で入力してください&quot;);
        }
    }

    // 単純なものは自動プロパティ
    public string Name { get; set; }
}

    自動プロパティのアクセス制御

    読み取り専用プロパティ

    public class Student
{
    public string StudentId { get; private set; }  // 外部からは読み取りのみ

    public Student(string studentId)
    {
        StudentId = studentId;  // コンストラクタ内では設定可能
    }
}

Student student = new Student(&quot;S001&quot;);
Console.WriteLine(student.StudentId);  // OK: 読み取り可能
student.StudentId = &quot;S002&quot;;            // エラー: 外部から設定不可

    初期値付き自動プロパティ

    public class Student
{
    public string Name { get; set; }
    public int Score { get; set; } = 0;     // 初期値を設定
    public bool IsActive { get; set; } = true;  // 初期値を設定
}

    使い分けの指針

    • 単純な値の保存 → 自動プロパティ
    • 検証や特別な処理が必要 → 従来のプロパティ
    • 外部から変更されたくない → { get; private set; }
    • 後から検証を追加する可能性 → 最初から従来のプロパティ

    実践例

    public class BankAccount
{
    // 単純な値 → 自動プロパティ
    public string AccountNumber { get; private set; }
    public string OwnerName { get; set; }

    // 検証が必要 → 従来のプロパティ
    private decimal balance;
    public decimal Balance
    {
        get { return balance; }
        private set 
        { 
            if (value &gt;= 0)
                balance = value;
            else
                throw new ArgumentException(&quot;残高は0以上である必要があります&quot;);
        }
    }
}

    オブジェクト指向の具体的な利点

    1. カプセル化の利点

    従来の方法（問題あり）

    // データが分散している
string[] studentNames = new string[100];
int[] studentAges = new int[100];
int[] studentScores = new int[100];

// 関連性が分からない
studentNames[0] = &quot;田中&quot;;
studentAges[0] = 20;
studentScores[1] = 85;  // ← 間違ったインデックス！

    オブジェクト指向（安全）

    // データと処理がまとまっている
Student student = new Student(&quot;田中&quot;, 20, 85);
// インデックスミスが起こらない

    2.再利用性の利点

    // 一度Studentクラスを作れば
Student student1 = new Student(&quot;田中&quot;, 20, 85);
Student student2 = new Student(&quot;佐藤&quot;, 19, 92);
Student student3 = new Student(&quot;山田&quot;, 21, 78);
// 何個でも簡単に作成できる

    3.保守性の利点

    // 成績計算ロジックを変更したい場合
// Studentクラスのメソッド1箇所を修正するだけで
// 全てのStudentオブジェクトに反映される
public string GetGrade()
{
    // ここを修正するだけ
    if (Score &gt;= 90) return &quot;A+&quot;;  // 基準を変更
    // ...
}

            AccountNumber = accountNumber;
            OwnerName = ownerName;
            Balance = 0;
        }

        {
            AccountNumber = accountNumber;
            OwnerName = ownerName;
            Balance = initialBalance;
        }

        {
            if (amount &gt; 0)
            {
                Balance += amount;
                Console.WriteLine($&quot;{amount:C}を入金しました。残高: {Balance:C}&quot;);
                return true;
            }
            else
            {
                Console.WriteLine(&quot;入金額は正の値である必要があります。&quot;);
                return false;
            }
        }

        public bool Withdraw(decimal amount)
        {
            if (amount &gt; 0 &amp;&amp; amount &lt;= Balance)
            {
                Balance -= amount;
                Console.WriteLine($&quot;{amount:C}を出金しました。残高: {Balance:C}&quot;);
                return true;
            }
            else if (amount &gt; Balance)
            {
                Console.WriteLine(&quot;残高が不足しています。&quot;);
                return false;
            }
            else
            {
                Console.WriteLine(&quot;出金額は正の値である必要があります。&quot;);
                return false;
            }
        }

            Console.WriteLine($&quot;名義人: {OwnerName}&quot;);
            Console.WriteLine($&quot;残高: {Balance:C}&quot;);
        }

        {
            if (Withdraw(amount))
            {
                targetAccount.Deposit(amount);

            }
        }

        {
            if (quantity &gt; 0 &amp;&amp; quantity &lt;= Stock)
            {
                Stock -= quantity;
                Console.WriteLine($&quot;{Name}: {quantity}個出荷しました。現在の在庫: {Stock}個&quot;);
                return true;
            }
            else
            {
                Console.WriteLine(&quot;在庫が不足しています。&quot;);

            Console.WriteLine($&quot;カテゴリ: {Category}&quot;);
        }

        public decimal CalculateTotalValue()
        {
            return Price * Stock;
        }

        public bool IsLowStock(int threshold = 10)
        {

        }
    }

    重要なポイント

    クラスとオブジェクトの基本概念

    • クラス：データと処理をまとめた設計図（型の定義）
    • オブジェクト：クラスから作られた実際のもの（インスタンス）
    • new演算子：クラスからオブジェクトを作成する命令
    • 独立性：同じクラスから作られたオブジェクト同士は独立している

    アクセス修飾子の使い分け

    • public：クラス外部からアクセス可能（外部インターフェース）
    • private：クラス内部からのみアクセス可能（内部実装の隠蔽）
    • 適切な保護：重要なデータはprivateにして、プロパティで安全にアクセス

    コンストラクタの理解

    • 初期化の専用メソッド：オブジェクト作成時に自動実行
    • メソッド名 = クラス名：戻り値の型は書かない
    • オーバーロード可能：引数の違いで複数定義
    • 必須の初期化：重要なデータを設定し忘れを防ぐ

    プロパティの活用

    • フィールドとプロパティの違い：直接アクセス vs 制御されたアクセス
    • getter/setter：値の取得と設定を制御
    • データ検証：不正な値の設定を防ぐ
    • 自動プロパティ：単純な場合は { get; set; } で簡潔に

    プロパティの使い分け

    • 自動プロパティ：単純な値の保存・取得のみ
    • 従来のプロパティ：検証や特別な処理が必要
    • 読み取り専用：{ get; private set; } で外部からの変更を防ぐ

    オブジェクト指向の実践的利点

    • カプセル化：関連データと処理をまとめて管理
    • データ保護：private + プロパティで安全なアクセス
    • 再利用性：同じクラスから複数のオブジェクトを作成
    • 保守性：修正箇所が明確で影響範囲が限定的

    設計の指針

    • 単一責任：1つのクラスは1つの概念を表現
    • 適切な命名：クラス名は名詞、メソッド名は動詞
    • 最小限の公開：必要最小限のメンバーのみpublicにする
    • 初期化の保証：コンストラクタで必要な初期化を確実に実行

    まとめ

    オブジェクト指向プログラミングの本質

    • 現実世界の概念をプログラムで表現：学生、口座、商品などをクラスとして定義
    • データと処理の統合：関連する情報と操作を1つのクラスにまとめる
    • 安全性の向上：適切なアクセス制御でデータを保護

    クラス設計の3つの要素

    • データ（フィールド・プロパティ）：オブジェクトが持つ情報
    • 処理（メソッド）：オブジェクトができる操作
    • 初期化（コンストラクタ）：オブジェクト作成時の設定

    これまでとの違い

    • 変数とメソッドが分散 → クラスで統合
    • データの直接操作 → プロパティによる制御
    • 初期化忘れのリスク → コンストラクタで保証
    • グローバルな影響 → オブジェクト単位の独立性

    実践的な活用パターン

    • エンティティクラス：学生、商品、口座などの「もの」を表現
    • データの集約：関連する複数の値をまとめて管理
    • ビジネスロジック：そのオブジェクト特有の処理を内包
    • 状態管理：オブジェクトの現在の状態を適切に保持

    開発効率の向上

    • コードの整理：関連機能がクラス単位でまとまる
    • バグの軽減：データと処理が密結合で整合性を保ちやすい
    • チーム開発：クラス単位で役割分担が明確
    • 機能拡張：既存クラスに新しいメソッドやプロパティを追加
