    🤔 なぜEntityとは別にフォーム用クラスが必要なのか？

    Week 8で作成したStudentクラスは、データベース表示には完璧でした。しかし、フォーム処理となると、少し事情が変わってきます。

    問題1: フォームとデータベースの項目の違い

    実際のユーザー登録フォームを考えてみましょう：

    📘 フォームとDBフィールドの比較

    ユーザーが入力するフォーム項目と、データベースの項目には違いがあります：

    フォーム項目 データベース項目 説明
    名前 name 同じ
    メールアドレス email 同じ
    年齢 age 同じ
    パスワード - フォームにはあるが、DBには暗号化後に保存
    パスワード確認 - フォームのみ（確認用）
    利用規約同意 - フォームのみ（必須チェック）
    - id DBにはあるが、新規登録時は自動生成
    - created_at DBにはあるが、システム自動設定

    この違いが、別々のクラスが必要な理由です！

    問題2: バリデーション要件の違い

    同じ「名前」フィールドでも、要件が異なります：

    // データベース用Entity
@Column(name = "name")
private String name;  // 単純な文字列格納

// フォーム用DTO  
@NotBlank(message = "名前は必須です")
@Size(min = 1, max = 50, message = "名前は1文字以上50文字以下で入力してください")
private String name;  // 入力チェック付き

    問題3: 責務の分離

    📘 エンティティとDTOの分離

    適切な設計では、以下のように責務を分離します：

    • Entity: データベースとの完全対応、永続化に特化
    • DTO: フォーム処理に特化、ユーザビリティ重視、バリデーション充実

    この分離により、保守性と可読性が大幅に向上します！

    🏗️ DTOパターンとは？

    DTOの定義

    DTO (Data Transfer Object) とは、データ転送専用のオブジェクトです。

    // DTO（Data Transfer Object）の例
public class StudentData {
    // フォーム処理に特化した設計
    private String name;
    private String email;
    private Integer age;
    private String password;        // フォーム専用
    private String confirmPassword; // フォーム専用
    private Boolean agreeTerms;     // フォーム専用
}

    DTOパターンの価値

    📘 DTOパターンの価値

    1. 責務の明確化: フォーム処理とデータベース処理を分離
    2. 保守性向上: 変更の影響範囲を限定
    3. 可読性向上: 各クラスの目的が明確
    4. テスタビリティ: 単体テストが書きやすい

    実務での活用

    実際の企業開発では、DTOパターンは標準的な設計手法です：

    • 大規模システム: 責務分離による複雑性管理
    • チーム開発: 担当領域の明確化
    • 長期保守: 変更に強い設計
    • 品質保証: テストしやすい構造

    バリデーションアノテーション解説

    📘 バリデーションアノテーション

    アノテーション 用途 例
    @NotBlank 必須入力（空白・null不可） 名前、メール等
    @NotNull null不可（空白は許可） 年齢等の数値
    @Email メール形式チェック メールアドレス
    @Size 文字数制限 パスワード等
    @Min/@Max 数値範囲制限 年齢等
    @AssertTrue true必須 利用規約同意
