    📘 Spring Boot進化記録

    1. Service層分離の必要性：Controller肥大化問題

    🚨 現在の問題点

    Week 9.7時点のControllerを詳しく見ると、以下の問題があります：

    @Controller
public class StudentController {
    @Autowired
    private StudentRepository studentRepository;

    @PostMapping("/students")
    public String registerStudent(@Valid @ModelAttribute StudentData studentData,
                                  BindingResult bindingResult,
                                  RedirectAttributes redirectAttributes) {
        if (bindingResult.hasErrors()) {
            return "students/register";
        }

        try {
            // 🚨 問題1: ビジネスロジックがControllerに混在
            // メールアドレス重複チェック
            if (studentRepository.existsByEmail(studentData.getEmail())) {
                redirectAttributes.addFlashAttribute("errorMessage", 
                    "このメールアドレスは既に使用されています");
                return "redirect:/students/register";
            }

            // 🚨 問題2: データベース操作が直接実行
            Student student = studentData.toEntity();
            studentRepository.save(student);

            // 🚨 問題3: 複雑なビジネスルールがController内に
            // 学年による専攻制限
            if (student.getGrade() == 1 && 
                !Arrays.asList("情報工学", "基礎教養").contains(student.getMajor())) {
                // 複雑な制御処理...
            }

            redirectAttributes.addFlashAttribute("successMessage", "学生を登録しました");
            return "redirect:/students";

        } catch (Exception e) {
            redirectAttributes.addFlashAttribute("errorMessage", "登録に失敗しました");
            return "redirect:/students/register";
        }
    }
}

    🔍 Controller肥大化による問題

    1. 単一責任原則の違反
    2. Controllerが「画面制御」と「ビジネスロジック」の両方を担当

    3. 1つのクラスが複数の責任を持つため、変更時の影響範囲が拡大

    4. テストの困難性

    5. Webリクエスト/レスポンスと一緒にビジネスロジックもテストする必要

    6. 単体テストが複雑になり、テストケースの作成・保守が困難

    7. 再利用性の低下

    8. ビジネスロジックがController内にあるため、他の画面やAPIで再利用不可

    9. 同じ処理を複数のControllerで重複実装するリスク

    10. 保守性の悪化

    11. ビジネスルール変更時に複数のControllerを修正する必要
    12. コードの可読性が低下し、新しいメンバーの理解が困難

    📘 Spring Boot MVC分離（Before/After）

    2. Service層による解決：MVCアーキテクチャの完全確立

    🏗️ Service層の役割

    Spring BootのService層は、ビジネスロジックを担当する専用レイヤーです：

    // MVCアーキテクチャの理想的な責任分離
Controller  : 画面制御（リクエスト受信、レスポンス返却、画面遷移）
Service     : ビジネスロジック（業務処理、データ変換、ビジネスルール）

@Service
public class StudentServiceImpl implements StudentService {

    @Autowired
    private StudentRepository studentRepository;

    @Override
    public void registerStudent(StudentData studentData) throws ServiceException {
        try {
            // ビジネスルール1: メールアドレス重複チェック
            validateEmailUnique(studentData.getEmail());

            // ビジネスルール2: 学年と専攻の整合性チェック

            // Entity変換とDB保存
            Student student = studentData.toEntity();
            studentRepository.save(student);

        } catch (Exception e) {

        }
    }

    private void validateEmailUnique(String email) throws ServiceException {

            throw new ServiceException("このメールアドレスは既に使用されています");
        }

    private void validateGradeAndMajor(int grade, String major) throws ServiceException {

        }
    }
