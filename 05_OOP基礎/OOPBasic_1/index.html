    🤔 手続き型プログラミングの「困った体験」を思い出そう

    Week 2で学生の成績管理システムを作った時のことを思い出してください。

    // Week 2で書いていたコード（手続き型）
public class Main {
    public static void main(String[] args) {
        // 学生データをバラバラに管理
        String[] studentNames = {"田中", "佐藤", "山田"};
        int[] scores = {85, 92, 78};
        String[] grades = new String[3];

        // 成績判定処理
        for (int i = 0; i < scores.length; i++) {
            if (scores[i] >= 90) {
                grades[i] = "A";
            } else if (scores[i] >= 80) {
                grades[i] = "B";
            } else if (scores[i] >= 70) {
                grades[i] = "C";
            } else {
                grades[i] = "D";
            }
        }

        // 結果表示
        for (int i = 0; i < studentNames.length; i++) {
            System.out.println(studentNames[i] + ": " + scores[i] + "点 (" + grades[i] + ")");
        }
    }
}

    📘 手続き型の問題点図解

    この書き方の「困った点」

    1. データがバラバラ：studentNames[0]、scores[0]、grades[0]が同じ学生のデータなのに別々の配列
    2. 管理が大変：学生を追加する時は3つの配列すべてを更新する必要がある
    3. 関係が見えない：コードを見ただけでは、どの配列同士が関連しているかわからない
    4. 再利用しにくい：この成績判定ロジックを他のプログラムで使いたい時に困る

    💡 オブジェクト指向という「解決策」

    オブジェクト指向プログラミング（OOP）は、上記の「困った点」を解決する プログラミングの考え方 です。

    オブジェクト指向の基本的な考え方

    「現実世界のモノ」をプログラムで表現する

    現実世界で「学生」を考えてみてください： - 学生には 名前 がある（データ） - 学生には 点数 がある（データ） - 学生は 成績を確認できる（動作） - 学生は 自己紹介できる（動作）

    オブジェクト指向では、この「学生」を ひとつのまとまり として扱います。

    📘 OOPの概念図

    🔄 思考の転換：手続き型 vs オブジェクト指向

    手続き型の思考

    「何をするか」を中心に考える

処理1 → 処理2 → 処理3 → 処理4
  ↓      ↓      ↓      ↓
データA データB データC データD

    オブジェクト指向の思考

    「誰が何をするか」を中心に考える

学生オブジェクト {
  データ：名前、点数
  動作：成績判定、自己紹介
}

教師オブジェクト {
  データ：担当科目
  動作：成績入力、レポート作成
}

    📘 手続き型とOOPの比較

    📝 実際に体験してみよう：まずは一歩目

    手続き型からオブジェクト指向への転換を、簡単な例で体験してみましょう。

    手続き型バージョン（従来の書き方）

    public class ProcMain {
    public static void main(String[] args) {
        // データと処理がバラバラ
        String name = "田中太郎";
        int score = 85;

        // 成績判定（処理）
        String grade;
        if (score >= 90) {
            grade = "A";
        } else if (score >= 80) {
            grade = "B";
        } else if (score >= 70) {
            grade = "C";
        } else {
            grade = "D";
        }

        // 表示（処理）
        System.out.println(name + ": " + score + "点 (" + grade + ")");
    }
}

    オブジェクト指向バージョン（新しい書き方）

    まずは簡単なStudentクラスを作ってみましょう：

    // Student.java
public class Student {
    // データ（属性）
    String name;
    int score;

    // 処理（動作）をデータと一緒にまとめる
    String getGrade() {
        if (score >= 90) {
            return "A";
        } else if (score >= 80) {
            return "B";
        } else if (score >= 70) {
            return "C";
        } else {
            return "D";
        }
    }

    void introduce() {
        System.out.println(name + ": " + score + "点 (" + getGrade() + ")");
    }
}

    // Main.java
public class Main {
    public static void main(String[] args) {
        // Studentオブジェクトを作成
        Student student = new Student();
        student.name = "田中太郎";
        student.score = 85;

        // 学生が自分で自己紹介する
        student.introduce();
    }
}

    // Studentクラス（今回作成）
public class Student {
    // 学生に関連するデータと処理をまとめた
    String name;    // データ
    int score;      // データ
    String getGrade() { ... }  // 処理
    void introduce() { ... }   // 処理
}

    どちらも「関連するものをまとめて整理している」という点で同じです！

    オブジェクト指向の3つのメリット

    1. 整理整頓：関連するデータと処理が一箇所にまとまる
    2. 再利用性：作ったクラスは何度でも使える
    3. 理解しやすさ：現実世界の「モノ」として考えられる

    💡 よくある質問

    Q1: StringUtilsクラスもオブジェクト指向だったのですか？

    A: 正確には「部分的に」です。StringUtilsは関連する機能をまとめるという点でOOP的ですが、staticメソッドのみで「オブジェクトを作る」という概念はありませんでした。今回学ぶのは、データと処理を一つのオブジェクトにまとめる「真のOOP」です。

    Q2: 手続き型とオブジェクト指向、どちらが良いのですか？

    A: どちらも価値のあるプログラミング手法です。小さなプログラムでは手続き型がシンプルで良い場合もあります。しかし、規模が大きくなったり、複数人で開発する場合は、オブジェクト指向の「整理整頓」の力が重要になります。

    Q3: 現実世界の「モノ」って具体的にどういうことですか？

    A: プログラムで表現したい対象を「モノ」として捉えることです。学生管理システムなら「学生」が一つのモノ、ECサイトなら「商品」「顧客」「注文」がそれぞれ一つのモノです。それぞれがデータ（属性）と動作（メソッド）を持ちます。
