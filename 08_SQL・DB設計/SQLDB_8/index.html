    レッスン7.7での成果

    レッスン7.7では、集約関数とGROUP BYを使った高度なデータ分析技術を習得しました：

    -- 学部別詳細分析（7.7で作成）
SELECT 
    s.department,
    COUNT(DISTINCT s.student_id) as total_students,
    COUNT(sc.score) as total_scores,
    ROUND(AVG(sc.score), 1) as avg_score,
    COUNT(CASE WHEN sc.score >= 90 THEN 1 END) as excellent_count,
    ROUND(COUNT(CASE WHEN sc.score >= 80 THEN 1 END) * 100.0 / COUNT(sc.score), 1) as success_rate
FROM student s
LEFT JOIN score sc ON s.student_id = sc.student_id
GROUP BY s.department
HAVING COUNT(sc.score) > 0
ORDER BY avg_score DESC;

    使用中のテーブル構造（7.7で構築）

    -- 学生基本情報テーブル
CREATE TABLE student (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL,
    age INT NOT NULL,
    department VARCHAR(100) NOT NULL,
    email VARCHAR(255),
    enrollment_date DATE,
    status ENUM('active', 'inactive', 'graduated') DEFAULT 'active'
);

-- 科目マスタテーブル
CREATE TABLE subject (
    id INT AUTO_INCREMENT PRIMARY KEY,
    subject_code VARCHAR(10) NOT NULL UNIQUE,
    subject_name VARCHAR(100) NOT NULL,
    credits INT NOT NULL,
    department VARCHAR(100)
);

-- 成績管理テーブル
CREATE TABLE score (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL,
    subject_code VARCHAR(10) NOT NULL,
    score INT NOT NULL,
    semester VARCHAR(20) NOT NULL,
    grade_letter ENUM('A', 'B', 'C', 'D', 'F'),
    FOREIGN KEY (student_id) REFERENCES student(student_id),
    FOREIGN KEY (subject_code) REFERENCES subject(subject_code)
);

    🤔 現在のテーブル設計の問題点は何でしょうか？

    レッスン7.7で様々な集約分析を行う中で、実はデータベース設計上の課題がいくつか潜んでいました。例えば：

    1. データの重複: 同じ学部名が複数の学生レコードに繰り返し記録される
    2. 更新の困難性: 学部名を変更する場合、複数箇所を更新する必要
    3. データ不整合のリスク: 入力ミスによる「情報科学部」と「情報科学科」の混在
    4. 拡張性の制限: 学部に関する詳細情報（設立年、建物など）を追加しにくい

    これらの問題を解決するのが、今回学ぶデータベース設計理論です！

    1. なぜデータベース設計が重要なのか？

    設計の良し悪しで決まること

    📘 データベース設計の比較

    良い設計の特徴： - データ整合性：矛盾のないデータ保持 - 保守性：変更・拡張が容易 - パフォーマンス：高速なデータ取得 - 可読性：理解しやすい構造

    悪い設計の問題： - データ重複：同じ情報の重複記録 - 更新異常：一部のデータのみ更新されるリスク - 削除異常：必要なデータの意図しない削除 - 挿入異常：関連データなしでは挿入できない

    企業開発での重要性

    なぜ企業でDB設計が重視されるのか？

1. データは企業の資産
   → 一度失ったデータは復旧困難

2. システムの長期運用
   → 5年、10年と使い続けるシステム

3. 複数システムでの共有
   → 会計システム、人事システム等での連携

4. 法的要件への対応
   → 個人情報保護、監査対応等

5. パフォーマンスへの影響
   → 設計ミスは後から修正困難

    2. 正規化理論：データの整理術

    正規化とは何か？

    正規化とは、データの重複や不整合を防ぐために、テーブル構造を段階的に整理していく手法です。

    📘 正規化プロセス

    段階別正規化の理解

    非正規形（正規化前）

    まず、正規化前の「問題のあるテーブル」を見てみましょう：

    -- 非正規形：全ての情報を1つのテーブルに詰め込んだ例
CREATE TABLE student_all_info (
    student_id VARCHAR(10),
    name VARCHAR(100),
    age INT,
    department VARCHAR(100),
    department_building VARCHAR(50),
    department_phone VARCHAR(20),
    email VARCHAR(255),
    subjects VARCHAR(500),  -- 「数学,物理,化学」のような形式
    scores VARCHAR(500),    -- 「85,92,78」のような形式
    semesters VARCHAR(500)  -- 「前期,前期,後期」のような形式
);

    問題点： - 一つのセルに複数の値（科目、点数等） - 科目を追加するたびにテーブル構造変更が必要 - 特定科目の平均点計算が困難

    第1正規形（1NF）

    定義: 「すべての属性が原子値（分割できない単一の値）であること」

    -- 第1正規形：繰り返しグループを排除
CREATE TABLE student_1nf (
    student_id VARCHAR(10),
    name VARCHAR(100),
    age INT,
    department VARCHAR(100),
    department_building VARCHAR(50),
    department_phone VARCHAR(20),
    email VARCHAR(255),
    subject VARCHAR(100),
    score INT,
    semester VARCHAR(20)
);

-- サンプルデータ
INSERT INTO student_1nf VALUES
('S001', '田中太郎', 20, '情報工学部', 'A棟', '03-1234-5678', 'tanaka@example.com', '数学', 85, '前期'),
('S001', '田中太郎', 20, '情報工学部', 'A棟', '03-1234-5678', 'tanaka@example.com', '物理', 92, '前期'),
('S001', '田中太郎', 20, '情報工学部', 'A棟', '03-1234-5678', 'tanaka@example.com', '化学', 78, '後期'),
('S002', '佐藤花子', 21, '情報工学部', 'A棟', '03-1234-5678', 'sato@example.com', '数学', 91, '前期');

    改善点: 原子値になった 残る問題: 学生情報と学部情報の重複

    第2正規形（2NF）

    定義: 「第1正規形 + 部分関数従属を排除」

    部分関数従属とは：「主キーの一部で決まってしまう属性」のこと

    -- 第2正規形：学生情報と成績情報を分離

-- 学生テーブル
CREATE TABLE student_2nf (
    student_id VARCHAR(10) PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    age INT NOT NULL,
    department VARCHAR(100) NOT NULL,
    department_building VARCHAR(50),
    department_phone VARCHAR(20),
    email VARCHAR(255)
);

-- 成績テーブル
CREATE TABLE score_2nf (
    student_id VARCHAR(10),
    subject VARCHAR(100),
    score INT NOT NULL,
    semester VARCHAR(20),
    PRIMARY KEY (student_id, subject, semester),
    FOREIGN KEY (student_id) REFERENCES student_2nf(student_id)
);

    改善点: 学生情報の重複排除 残る問題: 学部情報の重複（学部名変更時に複数レコード更新必要）

    第3正規形（3NF）

    定義: 「第2正規形 + 推移関数従属を排除」

    推移関数従属とは：「主キー以外の属性で決まってしまう属性」のこと

    -- 第3正規形：学部情報を独立したテーブルに分離

-- 学部テーブル
CREATE TABLE department_3nf (
    department_code VARCHAR(10) PRIMARY KEY,
    department_name VARCHAR(100) NOT NULL UNIQUE,
    building VARCHAR(50),
    phone VARCHAR(20),
    established_year INT
);

-- 学生テーブル（第3正規形）
CREATE TABLE student_3nf (
    student_id VARCHAR(10) PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    age INT NOT NULL,
    department_code VARCHAR(10) NOT NULL,
    email VARCHAR(255),
    enrollment_date DATE,
    FOREIGN KEY (department_code) REFERENCES department_3nf(department_code)
);

-- 科目テーブル
CREATE TABLE subject_3nf (
    subject_code VARCHAR(10) PRIMARY KEY,
    subject_name VARCHAR(100) NOT NULL,
    credits INT NOT NULL,
    department_code VARCHAR(10),
    FOREIGN KEY (department_code) REFERENCES department_3nf(department_code)
);

-- 成績テーブル（第3正規形）
CREATE TABLE score_3nf (
    score_id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL,
    subject_code VARCHAR(10) NOT NULL,
    score INT NOT NULL,
    semester VARCHAR(20) NOT NULL,
    grade_letter ENUM('A', 'B', 'C', 'D', 'F'),
    exam_date DATE,
    FOREIGN KEY (student_id) REFERENCES student_3nf(student_id),
    FOREIGN KEY (subject_code) REFERENCES subject_3nf(subject_code),
    UNIQUE KEY unique_score (student_id, subject_code, semester)
);

    改善点: - 学部情報の重複完全排除 - 学部情報変更時は1箇所のみ更新 - 新しい学部追加が容易 - データ整合性の向上

    3. 主キーと外部キー：データの整合性を守る仕組み

    主キー（Primary Key）の設計原則

    主キーとは

    • テーブル内で各行を一意に識別する属性または属性の組み合わせ
    • NULL値を持てない
    • 重複値を持てない
    • 値が変更されない（安定性）

    主キーの選択基準

    -- 良い主キーの例

-- 1. 自動増分ID（サロゲートキー）
CREATE TABLE student (
    id INT AUTO_INCREMENT PRIMARY KEY,  -- システムが自動で採番
    student_id VARCHAR(10) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL
);

-- 2. 業務上の自然なキー（ナチュラルキー）
CREATE TABLE subject (
    subject_code VARCHAR(10) PRIMARY KEY,  -- 業務で使用されるコード
    subject_name VARCHAR(100) NOT NULL
);

-- 3. 複合主キー（複数属性の組み合わせ）
CREATE TABLE score (
    student_id VARCHAR(10),
    subject_code VARCHAR(10),
    semester VARCHAR(20),
    score INT NOT NULL,
    PRIMARY KEY (student_id, subject_code, semester)  -- 3つの組み合わせで一意
);

    主キー設計の判断基準

    📘 主キー設計ガイド

    外部キー（Foreign Key）制約の重要性

    外部キーとは

    • 他のテーブルの主キーを参照する属性
    • 参照整合性を保証する仕組み
    • データの不整合を防ぐ

    外部キー制約の動作

    -- 外部キー制約の定義
CREATE TABLE score (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL,
    subject_code VARCHAR(10) NOT NULL,
    score INT NOT NULL,
    -- 外部キー制約の定義
    FOREIGN KEY (student_id) REFERENCES student(student_id)
        ON DELETE CASCADE    -- 学生削除時に成績も自動削除
        ON UPDATE CASCADE,   -- 学生ID変更時に成績の参照も自動更新
    FOREIGN KEY (subject_code) REFERENCES subject(subject_code)
        ON DELETE RESTRICT   -- 科目削除時はエラー（成績があるため）
        ON UPDATE CASCADE
);

    CASCADE等のオプション

    -- ON DELETE/UPDATE オプションの種類

-- CASCADE: 親のレコード操作に合わせて子も操作
-- RESTRICT: 子のレコードがある場合は親の操作を禁止
-- SET NULL: 親が削除/更新されたら子の外部キーをNULLに設定
-- NO ACTION: 何もしない（制約チェックは最後に実行）

-- 実用例
CREATE TABLE enrollment (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10),
    subject_code VARCHAR(10),
    enrollment_date DATE,
    FOREIGN KEY (student_id) REFERENCES student(student_id)
        ON DELETE CASCADE,  -- 学生退学時は履修記録も削除
    FOREIGN KEY (subject_code) REFERENCES subject(subject_code)
        ON DELETE RESTRICT  -- 履修者がいる科目は削除不可
);

    参照整合性の効果

    -- 外部キー制約なしの場合の問題例
INSERT INTO score (student_id, subject_code, score) 
VALUES ('S999', 'CS101', 85);  -- 存在しない学生IDでも挿入できてしまう

-- 外部キー制約ありの場合
-- Error 1452: Cannot add or update a child row: 
-- a foreign key constraint fails

    4. ER図（Entity-Relationship図）：設計の可視化

    ER図とは

    ER図は、データベースの構造を視覚的に表現する手法です： - Entity（エンティティ）: テーブルに相当 - Relationship（リレーションシップ）: テーブル間の関係 - Attribute（属性）: カラムに相当

    ER図の記号と読み方

    📘 ER図の記号一覧

    ステップ別ER図作成

    ステップ1: エンティティの抽出

    要件から「モノ」を抽出します：

    学生管理システムの要件：
- 学生の基本情報を管理する
- 学部・学科の情報を管理する  
- 科目の情報を管理する
- 学生の成績を記録する
- 履修登録を管理する

抽出されるエンティティ：
1. 学生（Student）
2. 学部（Department）
3. 科目（Subject）
4. 成績（Score）
5. 履修（Enrollment）

    ステップ2: 属性の決定

    各エンティティが持つ属性を決めます：

    学生（Student）:
- student_id (PK)
- name
- age
- email
- enrollment_date
- department_code (FK)

学部（Department）:
- department_code (PK)
- department_name
- building
- phone

科目（Subject）:
- subject_code (PK)
- subject_name
- credits
- department_code (FK)

成績（Score）:
- score_id (PK)
- student_id (FK)
- subject_code (FK)
- score
- semester
- grade_letter

    ステップ3: リレーションシップの定義

    エンティティ間の関係を定義します：

    リレーションシップ：

1. 学生 ←→ 学部
   - 関係：所属
   - カーディナリティ：多対一（N:1）
   - 一つの学部に複数の学生が所属

2. 科目 ←→ 学部  
   - 関係：開設
   - カーディナリティ：多対一（N:1）
   - 一つの学部が複数の科目を開設

3. 学生 ←→ 成績 ←→ 科目
   - 関係：受講・評価
   - カーディナリティ：多対多（M:N）
   - 学生は複数の科目を受講、科目は複数の学生が受講
   - 成績テーブルが中間テーブルとして機能

    完成したER図

    📘 完成版ER図

    ER図から物理設計へ

    ER図を基に実際のテーブル定義を作成します：

    -- ER図に基づく物理テーブル設計

-- 学部テーブル
CREATE TABLE department (
    department_code VARCHAR(10) PRIMARY KEY,
    department_name VARCHAR(100) NOT NULL UNIQUE,
    building VARCHAR(50),
    phone VARCHAR(20),
    established_year INT,
    INDEX idx_dept_name (department_name)
);

-- 学生テーブル
CREATE TABLE student (
    student_id VARCHAR(10) PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    age INT NOT NULL CHECK (age >= 15 AND age <= 100),
    email VARCHAR(255) UNIQUE,
    enrollment_date DATE NOT NULL,
    status ENUM('active', 'inactive', 'graduated') DEFAULT 'active',
    department_code VARCHAR(10) NOT NULL,
    FOREIGN KEY (department_code) REFERENCES department(department_code),
    INDEX idx_student_name (name),
    INDEX idx_student_dept (department_code),
    INDEX idx_student_status (status)
);

-- 科目テーブル
CREATE TABLE subject (
    subject_code VARCHAR(10) PRIMARY KEY,
    subject_name VARCHAR(100) NOT NULL,
    credits INT NOT NULL CHECK (credits > 0 AND credits <= 10),
    description TEXT,
    department_code VARCHAR(10),
    FOREIGN KEY (department_code) REFERENCES department(department_code),
    INDEX idx_subject_name (subject_name),
    INDEX idx_subject_dept (department_code)
);

-- 成績テーブル
CREATE TABLE score (
    score_id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL,
    subject_code VARCHAR(10) NOT NULL,
    score INT NOT NULL CHECK (score >= 0 AND score <= 100),
    semester VARCHAR(20) NOT NULL,
    grade_letter ENUM('A', 'B', 'C', 'D', 'F'),
    exam_date DATE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (student_id) REFERENCES student(student_id)
        ON DELETE CASCADE ON UPDATE CASCADE,
    FOREIGN KEY (subject_code) REFERENCES subject(subject_code)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    UNIQUE KEY unique_score (student_id, subject_code, semester),
    INDEX idx_score_student (student_id),
    INDEX idx_score_subject (subject_code),
    INDEX idx_score_semester (semester),
    INDEX idx_score_grade (grade_letter)
);

    5. 制約設計：データ品質の確保

    制約の種類と目的

    データベース制約は、データの整合性と品質を保証する仕組みです：

    PRIMARY KEY制約

    -- 主キー制約：一意性とNOT NULLを保証
CREATE TABLE student (
    student_id VARCHAR(10) PRIMARY KEY,  -- 重複不可、NULL不可
    name VARCHAR(100) NOT NULL
);

    FOREIGN KEY制約

    -- 外部キー制約：参照整合性を保証
CREATE TABLE score (
    student_id VARCHAR(10),
    subject_code VARCHAR(10),
    FOREIGN KEY (student_id) REFERENCES student(student_id),
    FOREIGN KEY (subject_code) REFERENCES subject(subject_code)
);

    UNIQUE制約

    -- 一意制約：重複を防ぐ（NULLは許可）
CREATE TABLE student (
    student_id VARCHAR(10) PRIMARY KEY,
    email VARCHAR(255) UNIQUE,  -- メールアドレスの重複防止
    phone VARCHAR(20) UNIQUE    -- 電話番号の重複防止
);

    NOT NULL制約

    -- NOT NULL制約：必須項目を保証
CREATE TABLE student (
    student_id VARCHAR(10) PRIMARY KEY,
    name VARCHAR(100) NOT NULL,        -- 名前は必須
    enrollment_date DATE NOT NULL,     -- 入学日は必須
    email VARCHAR(255)                 -- メールアドレスは任意
);

    CHECK制約

    -- CHECK制約：値の範囲や条件を制限
CREATE TABLE student (
    student_id VARCHAR(10) PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    age INT CHECK (age >= 15 AND age <= 100),     -- 年齢の範囲制限
    gpa DECIMAL(3,2) CHECK (gpa >= 0.0 AND gpa <= 4.0)  -- GPAの範囲制限
);

CREATE TABLE score (
    score_id INT AUTO_INCREMENT PRIMARY KEY,
    score INT CHECK (score >= 0 AND score <= 100),  -- 点数の範囲制限
    grade_letter ENUM('A', 'B', 'C', 'D', 'F'),     -- 評定の選択制限
    semester VARCHAR(20) CHECK (semester IN ('前期', '後期', '集中'))  -- 学期の制限
);

    制約の実践的な活用例

    -- 実用的な制約設計の例
CREATE TABLE enrollment (
    enrollment_id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL,
    subject_code VARCHAR(10) NOT NULL,
    enrollment_date DATE NOT NULL,
    withdraw_date DATE NULL,
    status ENUM('enrolled', 'completed', 'dropped') DEFAULT 'enrolled',

    -- 外部キー制約
    FOREIGN KEY (student_id) REFERENCES student(student_id)
        ON DELETE CASCADE ON UPDATE CASCADE,
    FOREIGN KEY (subject_code) REFERENCES subject(subject_code)
        ON DELETE RESTRICT ON UPDATE CASCADE,

    -- 複合一意制約：同じ学生は同じ科目に重複履修不可
    UNIQUE KEY unique_enrollment (student_id, subject_code),

    -- CHECK制約：退学日は履修日より後
    CHECK (withdraw_date IS NULL OR withdraw_date >= enrollment_date),

    -- CHECK制約：論理的整合性
    CHECK (
        (status = 'enrolled' AND withdraw_date IS NULL) OR
        (status IN ('completed', 'dropped') AND withdraw_date IS NOT NULL)
    )
);

    6. インデックス設計：パフォーマンス最適化の基礎

    インデックスとは

    インデックスは、データの検索を高速化するための仕組みです：

    -- インデックスなしの検索（全行スキャン）
SELECT * FROM student WHERE name = '田中太郎';
-- → 10万レコードあれば10万行すべてをチェック

-- インデックスありの検索（インデックススキャン）  
CREATE INDEX idx_student_name ON student(name);
SELECT * FROM student WHERE name = '田中太郎';
-- → インデックスで高速検索、数回のアクセスで発見

    インデックスの種類

    単一カラムインデックス

    -- よく検索されるカラムにインデックス作成
CREATE INDEX idx_student_name ON student(name);
CREATE INDEX idx_student_email ON student(email);
CREATE INDEX idx_score_semester ON score(semester);

    複合インデックス

    -- 複数カラムの組み合わせでよく検索される場合
CREATE INDEX idx_score_student_semester ON score(student_id, semester);
CREATE INDEX idx_score_subject_semester ON score(subject_code, semester);

-- このインデックスが有効な検索例：
SELECT * FROM score WHERE student_id = 'S001' AND semester = '前期';
SELECT * FROM score WHERE student_id = 'S001';  -- 先頭カラムのみでも有効

    ユニークインデックス

    -- 一意制約と高速検索を同時に実現
CREATE UNIQUE INDEX idx_student_email ON student(email);
CREATE UNIQUE INDEX idx_student_id ON student(student_id);

    インデックス設計の指針

    -- 学生管理システムでのインデックス戦略例

-- 主要テーブルの基本インデックス
CREATE TABLE student (
    student_id VARCHAR(10) PRIMARY KEY,  -- 自動でインデックス作成
    name VARCHAR(100) NOT NULL,
    department_code VARCHAR(10) NOT NULL,
    email VARCHAR(255) UNIQUE,           -- 自動でインデックス作成
    status ENUM('active', 'inactive', 'graduated') DEFAULT 'active',

    -- 外部キー用インデックス（JOIN性能向上）
    INDEX idx_student_dept (department_code),

    -- 検索用インデックス
    INDEX idx_student_name (name),
    INDEX idx_student_status (status),

    -- 複合インデックス（学部別のアクティブ学生検索）
    INDEX idx_dept_status (department_code, status)
);

CREATE TABLE score (
    score_id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL,
    subject_code VARCHAR(10) NOT NULL,
    semester VARCHAR(20) NOT NULL,
    score INT NOT NULL,
    grade_letter ENUM('A', 'B', 'C', 'D', 'F'),

    -- 外部キー用インデックス
    INDEX idx_score_student (student_id),
    INDEX idx_score_subject (subject_code),

    -- 検索・集計用インデックス
    INDEX idx_score_semester (semester),
    INDEX idx_score_grade (grade_letter),

    -- 複合インデックス（様々な検索パターンに対応）
    INDEX idx_student_semester (student_id, semester),
    INDEX idx_subject_semester (subject_code, semester),
    INDEX idx_semester_grade (semester, grade_letter)
);

    7. 実践演習：既存テーブルの改善設計

    現状分析：7.7で使用したテーブルの問題点

    レッスン7.7で使用したテーブル構造を設計の観点で分析してみましょう：

    -- 現在のテーブル構造（問題点あり）
CREATE TABLE student (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL,
    age INT NOT NULL,
    department VARCHAR(100) NOT NULL,    -- 問題：学部名の重複
    email VARCHAR(255),
    enrollment_date DATE,
    status ENUM('active', 'inactive', 'graduated') DEFAULT 'active'
);

-- 2. 学生テーブルの改善
CREATE TABLE student (
    student_id VARCHAR(10) PRIMARY KEY,    -- 主キーを業務キーに変更
    name VARCHAR(100) NOT NULL,
    age INT NOT NULL CHECK (age >= 15 AND age <= 100),
    department_code VARCHAR(10) NOT NULL,  -- 外部キーに変更
    email VARCHAR(255) UNIQUE,
    phone VARCHAR(20),
    enrollment_date DATE NOT NULL,
    graduation_date DATE NULL,
    status ENUM('active', 'inactive', 'graduated', 'leave') DEFAULT 'active',
    gpa DECIMAL(3,2) CHECK (gpa >= 0.0 AND gpa <= 4.0),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- 制約とインデックス
    FOREIGN KEY (department_code) REFERENCES department(department_code)
        ON UPDATE CASCADE ON DELETE RESTRICT,
    INDEX idx_student_name (name),
    INDEX idx_student_dept (department_code),
    INDEX idx_student_status (status),
    INDEX idx_student_gpa (gpa),

    -- 論理制約
    CHECK (graduation_date IS NULL OR graduation_date >= enrollment_date),
    CHECK (
        (status = 'graduated' AND graduation_date IS NOT NULL) OR
        (status != 'graduated' AND graduation_date IS NULL)
    )
);

-- 3. 科目テーブルの改善
CREATE TABLE subject (
    subject_code VARCHAR(10) PRIMARY KEY,
    subject_name VARCHAR(100) NOT NULL,
    credits INT NOT NULL CHECK (credits >= 1 AND credits <= 10),
    subject_type ENUM('required', 'elective', 'general') NOT NULL,
    prerequisite_subject VARCHAR(10) NULL,
    department_code VARCHAR(10),
    description TEXT,
    syllabus_url VARCHAR(500),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- 制約とインデックス
    FOREIGN KEY (department_code) REFERENCES department(department_code)
        ON UPDATE CASCADE ON DELETE SET NULL,
    FOREIGN KEY (prerequisite_subject) REFERENCES subject(subject_code)
        ON UPDATE CASCADE ON DELETE SET NULL,
    INDEX idx_subject_name (subject_name),
    INDEX idx_subject_dept (department_code),
    INDEX idx_subject_type (subject_type),
    INDEX idx_subject_credits (credits)
);

-- 4. 成績テーブルの改善
CREATE TABLE score (
    score_id INT AUTO_INCREMENT PRIMARY KEY,
    student_id VARCHAR(10) NOT NULL,
    subject_code VARCHAR(10) NOT NULL,
    academic_year INT NOT NULL,
    semester ENUM('spring', 'fall', 'summer', 'winter') NOT NULL,
    score INT CHECK (score >= 0 AND score <= 100),
    grade_letter ENUM('A+', 'A', 'B+', 'B', 'C+', 'C', 'D', 'F') NULL,
    exam_date DATE,
    exam_type ENUM('midterm', 'final', 'quiz', 'assignment', 'project') DEFAULT 'final',
    instructor_comment TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- 制約とインデックス
    FOREIGN KEY (student_id) REFERENCES student(student_id)
        ON UPDATE CASCADE ON DELETE CASCADE,
    FOREIGN KEY (subject_code) REFERENCES subject(subject_code)
        ON UPDATE CASCADE ON DELETE RESTRICT,
    UNIQUE KEY unique_score (student_id, subject_code, academic_year, semester, exam_type),
    INDEX idx_score_student (student_id),
    INDEX idx_score_subject (subject_code),
    INDEX idx_score_year_semester (academic_year, semester),
    INDEX idx_score_grade (grade_letter),
    INDEX idx_score_exam_date (exam_date),

    -- 論理制約
    CHECK (
        (score IS NULL AND grade_letter IS NULL) OR
        (score IS NOT NULL AND grade_letter IS NOT NULL)
    )
);

SELECT

    d.building,
    COUNT(s.student_id) as student_count
FROM department d
LEFT JOIN student s ON d.department_code = s.department_code

GROUP BY d.department_code, d.department_name, d.building
ORDER BY student_count DESC;
